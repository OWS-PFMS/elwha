# Changelog

All notable changes to Elwha are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Fixed

- **Shared `FlatSVGIcon` instances no longer cross-contaminate color across `ElwhaFab` / `ElwhaButton` / `ElwhaIconButton`** ([#197](https://github.com/OWS-PFMS/elwha/issues/197)) — each of these primitives installs a per-component `FlatSVGIcon.ColorFilter` to paint its glyph in the component's token-resolved foreground. Because `FlatSVGIcon` holds a single `colorFilter` field, constructing two components from the **same** `FlatSVGIcon` instance made the second install stomp the first, so both painted in the last-constructed component's color (and the consumer's own filter — e.g. `MaterialIcons`'s `Label.foreground` filter — was clobbered). Fixed by **defensively cloning** the icon (`new FlatSVGIcon(svg)`, which shares the parsed `SVGDocument` so the copy is cheap) before binding the per-component filter, so each component carries its own copy and the caller's shared instance is never mutated. The playgrounds dodged this by allocating a fresh `MaterialIcons.add()` per cell; consumers reusing one `Icon` across components no longer have to. Issue B from the report (a stale raster surviving `setColor`) does **not** reproduce on the bundled FlatLaf 3.2.5 — `paintIcon` applies the color filter live at paint time through a `GraphicsFilter` (there is no per-instance colored-raster cache to invalidate), so a `setColor` + `repaint()` already re-resolves the glyph on the next paint, as the `setColor` Javadoc already documents; no code change was needed for B and none was added. The same shared-icon pattern also exists in `ElwhaChip` and `ElwhaNavRailDestination` (more invasive — multiple call sites + theme-refresh re-apply paths); left for a scoped follow-up rather than expanded here, since this issue's Locations + acceptance enumerate the three button-family primitives. Smoketest: `com.owspfm.elwha.fab.playground.IconSharingColorDemo` (headless render + pixel-sample, exits non-zero on regression).
- **`ElwhaButton` Elevated shadow silhouette + offset regression** ([#218](https://github.com/OWS-PFMS/elwha/issues/218), same class as [#199](https://github.com/OWS-PFMS/elwha/issues/199)) — the Elevated variant's drop shadow rendered with squarer corners than the body (a darker band bulged past the rounded ends of a pill) and read as top-heavy rather than casting downward. Root cause is the `cornerRadiusPx()` convention foot-gun: `ElwhaButton` routes its body paint through the per-corner `CornerRadii` `SurfacePainter` overload (values are **real radii**), but `ShadowPainter.paint` takes a `RoundRectangle2D` **arcWidth** (corner diameter, 2× radius) like every other shadowed primitive feeds it (`ElwhaFab` / `ElwhaSurface` / `ElwhaCard`). `paintComponent` passed the bare `morphedRadii.largestPx()` radius, halving the shadow's corner radius and desyncing the shadow silhouette from the body. Fixed by doubling the radius (`largestPx() * 2`) at the `ShadowPainter` call so body and shadow agree on the same corner; a comment at the call site documents the convention to prevent re-breaking. Regression scoped to `ElwhaButton` — the other shadowed primitives use the int-arc (diameter) convention end-to-end and are unaffected.
- **`ElwhaNavigationRail` no longer auto-focuses the menu button on window activation** ([#256](https://github.com/OWS-PFMS/elwha/issues/256)) — `RailFocusTraversalPolicy.getDefaultComponent` returned `tabOrder[0]` (the menu button), so window activation and any programmatic Focus-default lookup parked a keyboard focus ring on the rail's chrome before the user did anything. Tab walking still enters the rail at the menu per design doc §10.2 (via `getFirstComponent`); only `getDefaultComponent` changes — it now returns the currently-selected destination (M3 tablist contract: focus enters at the selected tab, not at navigation chrome). The Phase 5 click-focus suppression (`setRequestFocusEnabled(false)` on slotted menu / FAB / trailing actions) was working correctly — the residual focus ring users perceived as "click grabs focus" was actually the leftover activation-cause focus that the click then refused to relinquish. Falls back to `getFirstComponent` if the rail has no destinations.
- **`ElwhaFab` Extended-form shadow regression** ([#199](https://github.com/OWS-PFMS/elwha/issues/199), regression introduced by [#198](https://github.com/OWS-PFMS/elwha/pull/198)) — the `getMaximumSize() { return getPreferredSize(); }` override added in the PR #198 code-review fixes caused the Extended FAB to render with an asymmetric dark concentration of shadow at the bottom-left corner. The override was preventative (no playground was actually stretching the FAB), and removing it restores the clean M3 elevation halo at all three sizes. `ElwhaIconButton`'s same override is unaffected because its `getPreferredSize()` returns the body alone — `ElwhaFab`'s includes the shadow reserve, and `max == preferred` on a halo-inclusive preferred disrupts Swing's paint pipeline. The exact mechanism is the subject of follow-up [#200](https://github.com/OWS-PFMS/elwha/issues/200).

### Changed

- **`ElwhaDialog` + `ElwhaFullScreenDialog` suppress the press ripple on dismiss action buttons** (epic [#288](https://github.com/OWS-PFMS/elwha/issues/288), story [#290](https://github.com/OWS-PFMS/elwha/issues/290)) — every dialog-wired action button (the Basic Dialog's confirm / alternate / cancel; the full-screen dialog's close ✕ + `confirmAction`) now calls `setRippleEnabled(false)`. The dialog's exit motion is the press feedback, and a live ripple would otherwise freeze mid-stroke on the one-time exit-fade snapshot. Removes the known-limitation that the full-screen dialog shipped with. **Consumer note:** the suppression persists on the button instance you pass as an action — re-enable with `setRippleEnabled(true)` if you reuse it outside the dialog.
- **Showcase nav rail dogfoods both dialog types** ([#292](https://github.com/OWS-PFMS/elwha/issues/292)) — the rail's trailing-action slot now carries two entries: a **(?)** help button opening a Basic `ElwhaDialog` ("how to use the Showcase"), and the existing **(i)** About button switched from `ElwhaDialog` to **`ElwhaFullScreenDialog`** (same About content; full-screen's headline + content slots, the tagline folded into the body, `contentMaxWidth(640)` to keep the longer-form content readable on a wide frame, dismissed by the ✕ / Esc). Demonstrates both M3 dialog primitives as live chrome on the library's own playground, with the Basic dialog as the short quick-decision surface and the full-screen dialog as the longer-form one. Library-internal change to `ElwhaShowcase` only.
- **`ElwhaDialog.DismissCause` promoted to top-level `com.owspfm.elwha.dialog.DismissCause`** (epic [#271](https://github.com/OWS-PFMS/elwha/issues/271), story [#276](https://github.com/OWS-PFMS/elwha/issues/276)) — **breaking, pre-1.0, no shims.** The dismiss-cause enum is no longer nested inside `ElwhaDialog`; it is a standalone top-level enum in the same `com.owspfm.elwha.dialog` package, shared by both dialog primitives (`ElwhaDialog` and the incoming `ElwhaFullScreenDialog`). Migration: replace `ElwhaDialog.DismissCause` with `DismissCause` (same package — add `import com.owspfm.elwha.dialog.DismissCause;` if referenced from elsewhere); the enum constants are unchanged. Done as part of extracting the shared overlay-host lifecycle into a new package-private `AbstractElwhaDialog` base (host resolution, `MODAL_LAYER` attach, dismiss/teardown, focus trap + restore, relayout, `MorphAnimator` plumbing) that both dialogs extend — a library-internal refactor with no other API change; the Basic Dialog is behavior-preserving (re-smoked via `DialogModalityDemo` / `DialogActionRowDemo` / the Showcase Dialog leaf + About dialog).
- **`ShadowPainter.paint` corner parameter renamed `cornerRadiusPx` → `arcWidthPx`** ([#285](https://github.com/OWS-PFMS/elwha/issues/285)) — the parameter was named `cornerRadiusPx` and documented as "the body's round-rect corner radius," but the implementation always treated the value as a `RoundRectangle2D` **arcWidth** (corner diameter, 2× radius). The name + Javadoc now match the behavior, and the doc explicitly flags that callers holding a real radius (e.g. `CornerRadii`, which stores radii) must double it. This is the latent footgun behind the recurring radius/diameter shadow regressions ([#199](https://github.com/OWS-PFMS/elwha/issues/199), [#218](https://github.com/OWS-PFMS/elwha/issues/218)). No behavioral change and no consumer break — `ShadowPainter` is library-internal (`public` only to cross the package boundary) and all callers pass positionally.
- **Showcase floating FAB shrinks on scroll** (epic [#205](https://github.com/OWS-PFMS/elwha/issues/205), story [#274](https://github.com/OWS-PFMS/elwha/issues/274)) — the Showcase's floating "FAB Workbench" FAB now dogfoods the `ElwhaFabAnchor` Phase 2 `ScrollResponse.SHRINK` behavior: it collapses Extended → Standard on scroll-down and expands back on scroll-up (M3 G33/G34). Because the content is a `CardLayout`, `showCard(...)` re-targets the anchor's scroll source to the active card's `JScrollPane` on every swap (cycling the response through `NONE` so each card starts un-shrunk); cards with no scroll pane leave the FAB Extended. Library-internal change to `ElwhaShowcase` only — no API change, `ElwhaFabAnchor` untouched.
- **Showcase floating FAB now placed by `ElwhaFabAnchor`** (epic [#205](https://github.com/OWS-PFMS/elwha/issues/205), story [#258](https://github.com/OWS-PFMS/elwha/issues/258)) — `ElwhaShowcase` drops its hand-rolled `addFloatingFab(JFrame)` `JLayeredPane` glue (manual bounds math + `ComponentListener` resize handler) and wraps its content in an `ElwhaFabAnchor` via `buildFloatingFabAnchor(JComponent)`, dogfooding the new primitive. The FAB now sits 16 dp off the visible-body edge (the prior recipe left it at 16 + shadow halo). The navigation rail still floats on the frame's own layered pane (leading edge), unchanged — the two placements use different panes and never contend. The floating FAB also gets `setRequestFocusEnabled(false)` to suppress the click-acquired focus ring, matching the rail's slotted controls (#256); the library-level fix (FAB paints its ring on keyboard focus only) is tracked in [#260](https://github.com/OWS-PFMS/elwha/issues/260). FAB design doc §15 rewritten to lead with `ElwhaFabAnchor` and keep the manual recipe as the under-the-hood reference. Library-internal change to `ElwhaShowcase` + docs only.
- **Showcase initial focus moves from the nav rail to the header mode toggle** ([#252](https://github.com/OWS-PFMS/elwha/issues/252)) — on app start, the Showcase now parks focus on the currently-selected segment of the Light/Dark/System `ElwhaButtonGroup` in the header bar instead of the rail. The rail's first destination is selected by default; auto-claiming focus there on startup read as a navigation prompt before the user had done anything, when the friendlier landing point is a header control the user can immediately act on with arrow keys. Library-internal change to `ElwhaShowcase` only.
- **`ElwhaNavigationRail` elevation 1 paints a real `ShadowPainter` drop shadow** (epic [#159](https://github.com/OWS-PFMS/elwha/issues/159), story [#249](https://github.com/OWS-PFMS/elwha/issues/249)) — **breaking, pre-1.0, no shims.** Replaces the Phase 2 placeholder inset trailing-edge gradient with the same `ShadowPainter` integration `ElwhaCard` / `ElwhaFab` / `ElwhaSurface` use — M3 key+ambient two-shadow stack at the elevation token. `paintComponent` calls `ShadowPainter.paint(g, bodyWidth, h, 0, elevation)` (rail body is a flat-edge rect, so `cornerRadiusPx = 0`). The rail's body is now distinct from its bounds: `bodyWidth()` reads `getWidth() - trailingShadowReserve()`, so a layered-pane host can widen the rail's `setBounds(...)` by `trailingShadowReserve()` to give the halo a clean landing zone — and the layout / surface fill / divider all stay scoped to the body silhouette. The Elwha Showcase's rail mount now sets `setElevation(1)` and widens bounds accordingly; `BorderLayout`-style hosts that can't reserve the inset see the halo silently clip on the body's trailing edge (documented trade-off, class Javadoc updated).
- **`ElwhaFab` morph paint refactored onto a new `ContentMorphPainter` helper** ([#223](https://github.com/OWS-PFMS/elwha/issues/223)) — the inlined icon-X interpolation, label cross-fade alpha (`(eased - 0.5) * 2` clamped to `[0, 1]`), and container-width interpolation that drive `ElwhaFab`'s Standard ↔ Extended morph paint pipeline are extracted into `com.owspfm.elwha.theme.ContentMorphPainter` — a stateless static helper parallel to `ShapeMorphPainter` exposing `iconX(fromX, toX, progress)`, `labelAlpha(progress)` (with the M3 `container-emphasized` 0.5 cross-fade inflection baked in, plus an overload accepting a caller-specified inflection), and `containerWidth(fromW, toW, progress)`. Each primitive takes an already-eased progress value so a single `MorphAnimator` feeding multiple parallel transitions can share one eased value across all three (matching today's FAB choreography exactly). `ElwhaFab.bodyWidthPx`, the icon-X math in `paintContent`, and the label-alpha math in `paintContent` all call into the helper now; the existing `formMorph` `MorphAnimator` field and the `Easing.EMPHASIZED` curve choice are unchanged. **No `ElwhaFab` public API change** — the refactor is paint-pipeline-internal and visually identical pre/post-refactor across all three sizes and six color styles (validated via `ElwhaFabMorphPlayground` and `ElwhaFabExtendedPlayground`). Companion smoketest at `com.owspfm.elwha.theme.playground.ContentMorphPlayground` visualizes the primitives in isolation for future consumers (Navigation Rail expand / collapse). FAB design doc §9.1 and §9.2 updated to reference the new helper alongside `ShapeMorphPainter`.
- **V3 Card playground action rows use `ElwhaButton`** ([#120](https://github.com/OWS-PFMS/elwha/issues/120), closes epic [#103](https://github.com/OWS-PFMS/elwha/issues/103)) — `ElwhaCardPlayground`'s gallery gains a "Card ↔ Button pairings" section demonstrating the M3 spec §3.3 pairing table (Elevated → Outlined + Filled, Filled → Text + Outlined, Outlined → Text + Filled-tonal), and the full-slot card's action row now uses `ElwhaButton` instead of raw `JButton`. The playground's remaining utility buttons (snippet "Copy", card-list add/remove) also move to `ElwhaButton` so the playground dogfoods the library end-to-end. The V1 → V3 migration doc gains an "Action-button pairing" pattern. No library API change — `ElwhaCardActions` already accepted any `JComponent`; this is the playground + docs demonstrating the full M3 action-row vocabulary.
- **`ElwhaCard` ripple paint migrated to `RipplePainter`** ([#121](https://github.com/OWS-PFMS/elwha/issues/121)) — `ElwhaCard.paintRipple` now delegates to the shared `RipplePainter.paint(...)` helper (extracted in [#115](https://github.com/OWS-PFMS/elwha/issues/115)) instead of the inline expanding-ellipse / fade math. Behavior is identical — the helper was extracted verbatim from this code. The card keeps ownership of the ripple animation `Timer` (the helper is stateless and consumes a caller-driven `progress`). Library-internal refactor only; no consumer-visible API change.
- **`ElwhaSurface` shadow paint migrated to `ShadowPainter`** ([#122](https://github.com/OWS-PFMS/elwha/issues/122), closes [#110](https://github.com/OWS-PFMS/elwha/issues/110)) — `ElwhaSurface.paintComponent` now delegates to `ShadowPainter.paint(...)` and `getInsets()` returns `ShadowPainter.shadowInsets(...)`. The per-instance `cachedShadow` + `cachedShadow{BodyW,BodyH,Arc,Elevation}` fields and the `shadowImageForCurrentState` helper are gone — replaced by `ShadowPainter`'s shared `(arc, elevation)`-keyed cache, 9-slice paint, and `SoftReference` eviction. **Resting-state visual change:** the new painter is the M3 key+ambient two-shadow stack, not the prior single-pass blur, so the chassis shadow reserve grows at any given elevation (at the V3 Card's MAX_ELEVATION=5 reserve: `(top=12, left=12, bottom=18, right=12)` → `(top=10, left=18, bottom=26, right=18)` — body width unchanged, halo region grows). Visible body content layout is unaffected.
- **`ElwhaCardMedia.painter` signature widened to `MediaPainter`** ([#21](https://github.com/OWS-PFMS/elwha/issues/21)) — **breaking, pre-1.0, no shims.** Was `ElwhaCardMedia.painter(Consumer<Graphics2D>)`; now `ElwhaCardMedia.painter(MediaPainter)` where `MediaPainter` is a new `@FunctionalInterface` with `void paint(Graphics2D g, int width, int height)`. Consumers receive the actual slot dimensions at paint time so they can scale / clip to the assigned area without closing over the component reference or reading `Graphics2D.getClipBounds`. Migration: change `g -> ...` to `(g, w, h) -> ...` and use `w` / `h` instead of hardcoded dimensions. Spec §5.2 amended.
- **`ElwhaCardDivider()` no-arg default flipped from `FULL` → `INSET`** (#18) — **breaking, pre-1.0, no shims.** Per M3 §1.7 contextual rule (now documented in spec §5.4), `FULL` pairs with a body-bottom expand text-link to mark the boundary between always-visible content and a collapsible hidden section; `INSET` is the within-body separator (metadata, footer rows, body → actions). `INSET` is the more common pattern, so the no-arg constructor now defaults to it. Consumers wanting the collapse-pair separator pass `new ElwhaCardDivider(DividerStyle.FULL)` explicitly.

### Removed

- **`ElwhaNavigationRail.ELEVATION_GRADIENT_PX` + `paintElevationGradient` helper** (epic [#159](https://github.com/OWS-PFMS/elwha/issues/159), story [#249](https://github.com/OWS-PFMS/elwha/issues/249)) — **breaking, pre-1.0, no shims.** The Phase 2 placeholder trailing-edge gradient is gone; its paint role is superseded by the real `ShadowPainter` integration above. `GradientPaint` import dropped from the rail along with the gradient constant and helper. No external consumer referenced these — they were package-private.
- **`MaterialPalettes.bundled()` removed** ([#153](https://github.com/OWS-PFMS/elwha/issues/153)) — **breaking, pre-1.0, no shims.** Superseded by the tier accessors `MaterialPalettes.primary()` and `MaterialPalettes.secondary()`; the flat bundled set is replaced by the Primary / Secondary tier split. No non-UI consumer existed.
- **`ElwhaSurface.setSuspendShadowRecompute(boolean)` removed** ([#122](https://github.com/OWS-PFMS/elwha/issues/122)) — **breaking, pre-1.0, no shims.** The protected hook that PR #110 added so `ElwhaCard`'s collapse animation could freeze shadow recompute mid-tween is gone. The redesigned `ShadowPainter` cache is keyed on `(arc, elevation)` only — body-size changes during the collapse tween no longer invalidate the cache, so the workaround is unnecessary at the root. `ElwhaCard.startCollapseAnimation` drops its start/stop calls. Subclasses that overrode the hook have no replacement and no longer need one.
- **`SurfacePainter.paintShadow` / `renderShadowImage` / `shadowInsets` removed** ([#122](https://github.com/OWS-PFMS/elwha/issues/122)) — the single-pass `ConvolveOp`-based shadow API is deleted; `ShadowPainter` is the only shadow path. `SurfacePainter`'s responsibility narrows to the round-rect fill + border body paint and the `bodyShape` geometry helper. Library-internal API only (the class was never part of the public surface) so this isn't a consumer-visible break.
- **`ElwhaCard` HORIZONTAL orientation deferred to v0.3.0** ([#112](https://github.com/OWS-PFMS/elwha/issues/112)) — the Phase-2 HORIZONTAL surface (`CardOrientation` enum, `ElwhaCard.setOrientation` / `getOrientation` / `PROPERTY_ORIENTATION`, `setLeadingColumn` / `setTrailingColumn`, private `TwoColumnLayout`, and the `addImpl` HORIZONTAL throw) is withdrawn before v0.2.0 ship. The shipped API created an asymmetric composition contract: VERTICAL used `card.add(...)` with the chassis owning layout, while HORIZONTAL forced `setLeadingColumn(JComponent)` / `setTrailingColumn(JComponent)` and threw on `add()` — same content needed two different construction paths, and the typed Layer-3 primitives (`ElwhaCardHeader`, `ElwhaCardActions`, `ElwhaCardDivider`) lost their auto-positioning behavior inside the trailing column. v0.3.0 re-enters HORIZONTAL under a unified `add(...)` API with typed partitioning (`ElwhaCardMedia` → leading column; everything else → trailing column running the same `VerticalCardLayout` rules) — see [spec §15.3](docs/research/elwha-card-v3-spec.md) for the design intent. Story [#89](https://github.com/OWS-PFMS/elwha/issues/89) (Phase-2 HORIZONTAL story) closed as superseded.

### Added

- **`ElwhaFab.getShadowInsets()`** ([#259](https://github.com/OWS-PFMS/elwha/issues/259)) — read-only accessor returning a copy of the shadow-halo reserve `getPreferredSize()` pads around the FAB's visible body (constant, sized for the hover elevation). Lets placement consumers back the halo out of the bounds when pinning the body to a spec margin instead of duplicating the private `HOVER_ELEVATION` constant and recomputing `ShadowPainter.shadowInsets(4)` themselves. `ElwhaFabAnchor` and `ElwhaNavigationRail` both drop their mirrored `FAB_HOVER_ELEVATION` copy and read this accessor. Surfaced during [#258](https://github.com/OWS-PFMS/elwha/issues/258). Read-only geometry exposure — no behavior change, purely additive.
- **`ElwhaIconButton` press shape-morph (M3 Expressive)** ([#295](https://github.com/OWS-PFMS/elwha/issues/295)) — pressing an icon button now animates its corner radius toward the opposite shape and snaps back on release, the icon button's analogue of `ElwhaButton`'s `pressMorph` ([#176](https://github.com/OWS-PFMS/elwha/issues/176)) and part of the same M3 Expressive shape-morph family. Driven by a `MorphAnimator` at the §3-pinned `short3` (150ms) press timing with the same per-direction easing as the button (`emphasized.decelerate` in / `emphasized.accelerate` out); reuses the shared `ShapeMorphPainter`. Wired to mouse press/release (and press-cancel via exit / focus-loss) and pulsed for keyboard `Space`/`Enter` activation. Honors the global reduced-motion flag (snaps to the end state). **Three deliberate departures from `ElwhaButton`'s tuning**, because the icon button has no companion press-*width* morph to carry the cue: the morph magnitude is a *ratio* of the button's max radius (40%, floored at a 4px delta) rather than a flat 4px so it reads across all sizes; there is **no "leave pill/circle corners alone" guard** — a round (`ShapeScale.FULL`) icon button morphs its corner, which is the whole point (a quick click on a ripple-suppressed dismiss ✕, [#290](https://github.com/OWS-PFMS/elwha/issues/290), would otherwise confirm nothing once the ripple is gone); and the morph is **bidirectional** — round-ish shapes square up while an already-square (`ShapeScale.NONE`) shape rounds *out*, so the press is visible at every shape rather than dying on the square end. Coexists with the press/hover/selected state layers and the press ripple — neither is disturbed. No API change: default-on, paint-pipeline-internal (the resting render is byte-identical — `paintComponent` now routes through the per-corner `SurfacePainter`/`RipplePainter` overloads, converting the int-arc shape value to a real radius via `/2` to match the prior int-arc paint). Smoketest: `com.owspfm.elwha.iconbutton.playground.IconButtonPressMorphPlayground`, whose "Ripple" toggle (driving the `setRippleEnabled` added by epic [#288](https://github.com/OWS-PFMS/elwha/issues/288)) confirms the morph fires identically with the ripple on or off — it is an independent paint layer.
- **`setRippleEnabled(boolean)` / `isRippleEnabled()` on `ElwhaButton` + `ElwhaIconButton`** (epic [#288](https://github.com/OWS-PFMS/elwha/issues/288), story [#289](https://github.com/OWS-PFMS/elwha/issues/289)) — a fluent toggle gating the press / touch ripple (the `RipplePainter` circle seeded on press or keyboard activation). Disabling clears any in-flight ripple immediately. It gates **only** the press ripple — the pressed state-layer darken and the connected-group width-ripple ([#176](https://github.com/OWS-PFMS/elwha/issues/176)) are unaffected. Default `true`, so existing buttons are unchanged. Added for hosts whose own motion is the press feedback (see the dialog wiring in [#290](https://github.com/OWS-PFMS/elwha/issues/290)). Smoketest: `RippleToggleDemo`.
- **`ElwhaFullScreenDialog` — M3 Full-screen Dialog primitive** (epic [#271](https://github.com/OWS-PFMS/elwha/issues/271), Phase 1 — stories [#276](https://github.com/OWS-PFMS/elwha/issues/276)–[#280](https://github.com/OWS-PFMS/elwha/issues/280)) — a token-themed modal dialog (`com.owspfm.elwha.dialog.ElwhaFullScreenDialog`) that fills the host frame edge-to-edge for longer-form input flows (multi-field forms) and narrow frames where a centered Basic Dialog would be cramped, with a fluent `Builder`. A **sibling** of `ElwhaDialog`, not a variant flag — the two share the extracted `AbstractElwhaDialog` overlay-host lifecycle. The container is `ColorRole.SURFACE` / `ShapeScale.NONE` (0dp) / no shadow (it *replaces* the page rather than floating over it), so **there is no scrim** — the surface physically covers all app content and the shared focus trap keeps keyboard focus inside. Content sits in a max-560dp column centered horizontally (full width on a narrow frame) with 24px padding. A 56px inline **top app bar** carries the leading close affordance (an `ElwhaIconButton` + the new `MaterialIcons.close()` glyph, dismissing `CANCEL`), the start-aligned `TypeRole.TITLE_LARGE` headline, and an optional trailing confirm text button (dismissing `CONFIRM` after the consumer's listener), with an optional 1px `OUTLINE_VARIANT` divider that also appears when the content scrolls. Content is wrapped in a vertical-only scroll pane (the app bar stays pinned). `Esc` → close, `Enter` → confirm (hand-wired, since `ElwhaButton` is not a `JButton`). `show(Component)` is non-blocking and reports the outcome via `onClose(Consumer<DismissCause>)`; `dismiss()` closes programmatically; `renderPreview()` renders the surface non-modally for galleries / docs. A11y: surface reports `AccessibleRole.DIALOG` with the headline as accessible name, initial focus lands on the first content field (it's an input flow), focus is trapped while shown and restored to the trigger on close. RTL mirrors the app bar (close trailing / confirm leading / headline right-aligned). Entrance/exit motion (Phase 2, [#281](https://github.com/OWS-PFMS/elwha/issues/281)) slides the container up from a small downward offset (≤48px / 8% of frame height) and fades it `0→1` on open via the shared `MorphAnimator` (emphasized-decelerate, 300ms), reversing (slide-down + fade-out) on close, with the same full-size-snapshot reuse the Basic Dialog uses; reduced motion snaps to the end state. Builder: `headline` / `content` / `confirmAction` / `showDivider` / `dismissibleByEsc` / `onClose`. The Elwha Showcase's "Dialog" leaf (Phase 3, [#282](https://github.com/OWS-PFMS/elwha/issues/282)/[#283](https://github.com/OWS-PFMS/elwha/issues/283)) gains an "Open full-screen dialog" Workbench trigger (live on the frame, with confirm / divider / Esc / reduced-motion controls + a sample form) and a static `renderPreview()` snapshot in its Gallery. Action buttons (the close ✕ and `confirmAction`) suppress their press ripple via `ElwhaButton.setRippleEnabled(false)` so it can't freeze on the exit-fade snapshot (epic [#288](https://github.com/OWS-PFMS/elwha/issues/288), story [#290](https://github.com/OWS-PFMS/elwha/issues/290)). ### Fixed

- **`ElwhaCard` preferred size is self-consistent before first layout** ([#157](https://github.com/OWS-PFMS/elwha/issues/157)) — `VerticalCardLayout.preferredLayoutSize` now computes the card's natural body width first and measures the width-sensitive children (cover-fit `ElwhaCardMedia`, wrap-row `ElwhaCardActions`) against it, so the returned `(width, height)` pair is self-consistent: a parent that honors the card's preferred width gets exactly that height with no re-layout settling cycle. Previously, before the chassis had been laid out once, a painter-backed media slot fell back to its hardcoded 320 dp intrinsic width and the card reported a too-short preferred height — a card wider than 320 dp clipped its header / supporting text off the bottom. Surfaced by the Elwha Showcase Card Workbench ([#144](https://github.com/OWS-PFMS/elwha/issues/144)), which drops its fixed-media-height workaround as a result.
- **`OUTLINED` `ElwhaCard` keeps a continuous border behind an edge-bleed media slot** ([#157](https://github.com/OWS-PFMS/elwha/issues/157)) — the outline is now painted above the children, in `paintChildren`, for every `OUTLINED` card. An edge-bleed `ElwhaCardMedia` slot fills the card to the chassis edge with an opaque cover and previously occluded the border along the card's top and upper-side edges, leaving the card half-outlined; painting the border last makes it continuous regardless of media. The resting, focused (PL-8), and disabled (PL-10) outline treatments now share the paint-above-children path.
- **Card corners no longer stair-step where an edge-bleed media slot meets the chassis curve** ([#157](https://github.com/OWS-PFMS/elwha/issues/157)) — `ElwhaSurface.paintChildren` clipped children to the rounded body with `Graphics2D.clip()`, and Java2D never antialiases a clip boundary, so an opaque edge-to-edge child (an `ElwhaCardMedia` cover) rendered jagged, aliased corners against the smooth antialiased chassis fill — conspicuous on a HiDPI display. Children are now rendered into a device-resolution offscreen buffer and composited back through an antialiased rounded-rect mask (soft clipping), so every child's corner curve matches the chassis exactly.

The previous 14 px was chip-specialized; Material Symbols are designed at the 20-dp optical-size axis, so 24 px keeps them at their design-intended visual weight for icon buttons and toolbars. **Breaking visible change** for any consumer calling the no-arg `MaterialIcons.foo()` form — icons will render larger by default. Internally, the 13 chip-context call sites in `ElwhaChipList` and `ElwhaChipPlayground` are pinned to an explicit `(14)` so chip visuals are preserved; chip-icon sizing will be revisited during the ElwhaChip V2 refresh.
- **`FlatPill` renamed to `ElwhaChip`** ([epic #27](https://github.com/OWS-PFMS/elwha/issues/27)) — aligns the component name with Material Design's chip taxonomy. **Breaking change** for any consumer importing the old names, even though no consumer has shipped yet against this lib. The rename is mechanical and 1:1; no behavior changes.
    - **Package:** `com.owspfm.ui.components.pill` → `com.owspfm.elwha.chip`; `pill.list` → `chip.list`.
    - **Classes:** every `FlatPill*` and `Pill*` type renamed (`FlatPill` → `ElwhaChip`, `FlatPillList` → `ElwhaChipList`, `FlatPillPlayground` → `ElwhaChipPlayground`, `FlatPillDemo` → `ElwhaChipDemo`, `PillVariant` → `ChipVariant`, `PillInteractionMode` → `ChipInteractionMode`, `PillListModel` / `DefaultPillListModel` → `ChipListModel` / `DefaultChipListModel`, `PillSelectionModel` / `DefaultPillSelectionModel` → `ChipSelectionModel` / `DefaultChipSelectionModel`, `PillSelectionMode` → `ChipSelectionMode`, `PillSelectionEvent` / `PillSelectionListener` → `ChipSelectionEvent` / `ChipSelectionListener`, `PillAdapter` → `ChipAdapter`, `PillReorderEvent` / `PillReorderListener` → `ChipReorderEvent` / `ChipReorderListener`, `PillListDataEvent` / `PillListDataListener` → `ChipListDataEvent` / `ChipListDataListener`).
    - **UIManager keys:** every `"FlatPill.*"` string value renamed to `"ElwhaChip.*"` (`ElwhaChip.background`, `ElwhaChip.hoverBackground`, `ElwhaChip.padding`, `ElwhaChip.arc`, `ElwhaChip.warmAccent`, `ElwhaChip.foreground`, …). Java constant names (`K_*`, `STYLE_PROPERTY`) are unchanged; only the string values move. The client-property key `"FlatPill.style"` is now `"ElwhaChip.style"`, and the ad-hoc `"FlatPill.removeIcon"` override slot used by the playground is now `"ElwhaChip.removeIcon"`.
    - **Consumer migration:** any consumer that previously overrode `UIManager.put("FlatPill.*", ...)` at app startup must update its overrides to the `ElwhaChip.*` namespace. Imports of `com.owspfm.ui.components.pill.*` must update to `com.owspfm.elwha.chip.*`. Tracked downstream in [OWS-PFMS/OWS-Local-Search-GUI#258](https://github.com/OWS-PFMS/OWS-Local-Search-GUI/issues/258).
    - **History:** the component was originally named `FlatPill` because its shape is a capsule ("pill"); the Material industry-standard name is "chip" for interactive token elements (input / choice / filter / action chips). Pre-flight inventory of every renamed touchpoint lives at `docs/research/flatpill-to-flatchip-rename-inventory.md`.
- **Naming convention switched to Google Java Style** ([epic #5](https://github.com/OWS-PFMS/elwha/issues/5)). Every `my*`-prefixed instance field and every `the*`-prefixed parameter / local was renamed to its unprefixed equivalent across the source tree. Public API parameter names move with this change — Java does not bind to parameter names at the call site, but Javadoc `@param` references and consumers reading reflection metadata will see the new names. No method signatures, class names, package names, or behavior changed.
- **`ChipVariant.GHOST` does not participate in selection rendering** ([#50](https://github.com/OWS-PFMS/elwha/issues/50), rebuild doc §9 Q2 amendment) — GHOST is Elwha's equivalent of M3's Text button (the lowest of M3's five emphasis levels), and M3 doesn't define a selected state at that emphasis level. Resolves a visual collapse surfaced during PR #46 smoketest: under the preemptive `PRIMARY` border swap on selected, `GHOST + SELECTED` and `OUTLINED + SELECTED` rendered identically (both: no fill + 12 % overlay + 1 px `PRIMARY` border), and `GHOST + SELECTED` additionally read as a faded `FILLED` chip — defeating GHOST's "no surface unless interacting" intent. The `selected` field and `PROPERTY_SELECTED` events are preserved on the API surface (so `setVariant(...)` later to a non-GHOST variant resumes rendering the selected state), but the rendering pipeline ignores `selected` while variant is `GHOST` — no `SELECTED` state-layer overlay, no border swap. Consumers needing a togglable chip with a visible selected state should use `OUTLINED` (M3's default treatment for filter chips). `ChipVariant × ChipInteractionMode` orthogonality is preserved at the API level — only the visual contract changes.

### Removed

- **`ChipVariant.WARM_ACCENT`** (epic #31) — never an M3 concept and bundled a color choice into what should be a treatment axis. The secondary-container look is now `FILLED` + `setSurfaceRole(SECONDARY_CONTAINER)` — color and treatment cleanly orthogonal.
- **`ElwhaChip.*` UIManager namespace** (epic #31) — the entire ~12-key namespace (`ElwhaChip.background`, `borderColor`, `arc`, `padding`, `hoverBackground`, `pressedBackground`, `selectedBackground`, `selectedBorderColor`, `focusColor`, `disabledBackground`, `warmAccent`, `foreground`) and the `K_*` public constants are deleted. App-wide chip theming now happens through the installed palette + the variant's role declarations; per-instance happens through the typed setters. The `"ElwhaChip.style"` client-property escape hatch is also gone — typed setters cover what it did.
- **`ElwhaChip` raw-typed setters** (epic #31) — `setSurfaceColor(Color)`, `setForegroundColor(Color)`, `setCornerRadius(Integer)`, `setBorderColor(Color)`, `setPadding(Insets)`, `setPadding(int, int)`. Replaced by `setSurfaceRole(ColorRole)`, `setShape(ShapeScale)`, `setPadding(SpaceScale, SpaceScale)`. `setBorderWidth(int)` is **kept** — stroke width is genuine geometry with no token equivalent.
- **`ElwhaChip` foreground override** (epic #31) — no per-instance foreground setter. Foreground is always the `on`-pair of the effective surface role, so an independent override would re-introduce unpaired surface/foreground.
- **`ElwhaChip` auto-contrast machinery** (epic #31) — `resolveForegroundColor()`'s luma branch, `AUTO_FG_DARK` / `AUTO_FG_LIGHT`, `isLight`, `effectiveSurfaceForContrast`. The token system's `on`-pairing makes them unnecessary.
- **`ElwhaChipDemo`** (epic #31) — the minimal smoke-test demo. Redundant with `ElwhaChipPlayground` after the playground rework; the playground now covers the same matrix plus the live-list surface.

### Fixed

- **`DefaultChipSelectionModel.setSelected` operated on the caller's list, not the model's** ([#149](https://github.com/OWS-PFMS/elwha/issues/149)) — the method's `selected` parameter shadowed the `selected` field (the model's `LinkedHashSet`), so `setSelected` cleared and refilled the *caller's* list while leaving the model untouched, and threw `UnsupportedOperationException` outright when handed an immutable list. `ElwhaChipList` passes `List.of(...)` from `ensureMandatorySelection` and every click-to-select / range-select path, so `SINGLE_MANDATORY` selection and range selection were broken for any populated list. Fixed by qualifying the field writes with `this.`. Surfaced by the Showcase's new `Component | Surface` chip switcher — the first `SINGLE_MANDATORY` list built over a pre-populated model.
- **`SurfacePainter` stroke geometry for borders wider than 1 px** (surfaced by smoketesting [#43](https://github.com/OWS-PFMS/elwha/issues/43)) — the painter centered the stroke on a path inset by `0.5 px` (correct for a 1 px AA line on the half-pixel grid), so any wider stroke extended outside the component bounds and got clipped: at width 8 only ~4.5 px of fuzzy inner stroke was visible instead of a crisp outer ring. Fixed by insetting the stroke centerline by `borderWidth / 2` and shrinking the corner diameter by `borderWidth` so the stroke's outer edge tracks the fill's corner exactly. Fill now extends to the full component bounds for clean AA at the corners. Latent under `ElwhaChip` (max stroke 2 px, artifact below the visibility threshold); newly user-visible via `ElwhaSurface.setBorderWidth(int)`.
- **`ElwhaChip` text baseline drift when leading affordance is hidden** ([#49](https://github.com/OWS-PFMS/elwha/issues/49)) — chips with no visible leading affordance but a visible trailing icon read with the text sitting ~2px high relative to the trailing icon's centerline. Root cause: stock `FlowLayout` skipped invisible children for row-height computation, so the leading cluster reported a shorter preferred height than the trailing button's hit-target; when `BorderLayout` stretched the cluster to match, the slack landed below the row instead of being split above and below. Fixed by an internal `BaselineCenteringFlowLayout` that floors the row height at `BUTTON_MIN_HIT_TARGET` and re-centers the row after the parent's layout pass. Predates epic #31; surfaced by smoketesting the chip rebuild.
- **Bundled `MaterialPalettes.baseline()` regenerated from a fresh Material Theme Builder export** ([#47](https://github.com/OWS-PFMS/elwha/issues/47)) — the prior baseline scheme produced muddy / low-legibility chip surfaces, and additionally collapsed the secondary role onto primary (`schemes.light.primary == schemes.light.secondary`), nullifying the primary/secondary emphasis hierarchy M3 components are designed around. New palette is a clean single-seed M3 scheme (`#6750A4`); secondary and tertiary tonal palettes are derived by canonical hue-rotation, so `primary` / `secondary` / `tertiary` (and their containers) are distinct in both light and dark. All 49 `ColorRole` keys now come directly from `schemes.light` / `schemes.dark` (no derived roles). Raw export archived at `docs/research/baseline-palette-mtb-export-2026-05-15.json`.

## [0.1.0] — 2026-05-12

Initial release. Library extracted from the [OWS-Local-Search-GUI](https://github.com/OWS-PFMS/OWS-Local-Search-GUI) project ([epic #231](https://github.com/OWS-PFMS/OWS-Local-Search-GUI/issues/231)).

### Added

- **`ElwhaCard`** — theme-aware card primitive with header, body, surface variants, hover/pressed/selected states, optional collapse/expand, leading icon, trailing actions.
- **`ElwhaCardList<T>`** — list of `ElwhaCard` items: selection, drag-to-reorder, filter, sort, orientation modes (vertical / horizontal / wrap / grid).
- **`FlatPill`** — compact pill primitive: text + optional leading icon + optional trailing action button, themeable via UIManager keys (`FlatPill.*`), auto-contrast foreground via BT.601 luma.
- **`FlatPillList<T>`** — list of `FlatPill` items with selection modes (`NONE` / `SINGLE` / `SINGLE_MANDATORY` / `MULTIPLE`), drag-to-reorder, pinned-partition + anchored modes (`MovementMode` enum), icon affordances (`IconAffordance` enum).
- **`ElwhaList<T>`** — shared cross-cutting interface implemented by both `ElwhaCardList` and `FlatPillList` (orientation, gap, padding, empty / loading state, filter, sort).
- **`MaterialIcons`** — helper that exposes 17 Material Symbols SVGs (Rounded / 400 / fill0 / 20px) via `FlatSVGIcon` with a theme-aware color filter.
- **`ElwhaCardPlayground`** — interactive playground for the card family (embeds `ElwhaCardListShowcase` + `GalleryPanel` panels for the list view, single-card variants, and cycle/cursor examples).
- **`FlatPillPlayground`** — interactive playground for the pill list family, including a LAF tweak panel for live UIManager-key experimentation.

### Notes

- API is **not yet stable** at 0.1.0. Breaking changes between minor versions are expected until 1.0.0.
- Two breaking-change refactors are tracked and gated on this release:
  - `FlatPill` → `ElwhaChip` rename (taxonomy alignment with Material)
  - `ElwhaList<T>` extension to share selection + drag-reorder surface across both list families
- 1.0.0 will follow those landings.
