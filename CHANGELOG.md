# Changelog

All notable changes to Elwha are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **ElwhaIconButton — token-native M3 icon-button primitive** ([epic #45](https://github.com/OWS-PFMS/elwha/issues/45)) — a square, icon-only button under the new `com.owspfm.elwha.iconbutton` package with four M3 emphasis variants (`FILLED` / `FILLED_TONAL` / `OUTLINED` / `STANDARD`), a `CLICKABLE` / `SELECTABLE` interaction axis, and a declarative `setIcons(resting, selected)` toggle pattern. The third token-native component built on `SurfacePainter`, after `ElwhaChip` (#31) and `ElwhaSurface` (#43); subsumes the playground's hand-rolled `pushPin ↔ pushPinFilled` borderless-toggle pattern. Decisions and the deliberate M3 divergences (uniform state-layer overlay vs M3's per-variant surface swap, single 40 dp size, no shape morphing): [`docs/research/elwha-icon-button-design.md`](docs/research/elwha-icon-button-design.md).
    - **Design doc** ([#55](https://github.com/OWS-PFMS/elwha/issues/55)) — locks the variant table (`FILLED` → `PRIMARY`, `FILLED_TONAL` → `SECONDARY_CONTAINER`, `OUTLINED` → transparent + `OUTLINE`, `STANDARD` → transparent + no border), the foreground-resolution rule (`on`-pair of the effective surface, falling back to `ON_SURFACE_VARIANT`, with the single `STANDARD + selected` → `PRIMARY` exception), the defaults (STANDARD + CLICKABLE + `ShapeScale.FULL` + 40 dp container + 24 dp icon), and §10's explicit table of every place Elwha intentionally diverges from the M3 icon-button spec.
    - **`ElwhaIconButton` class + `IconButtonVariant` + `IconButtonInteractionMode`** ([#56](https://github.com/OWS-PFMS/elwha/issues/56)) — fluent setters return `this`; paint delegates to `SurfacePainter` (no round-rect code in this class); icon tinting via the same `FlatSVGIcon.ColorFilter` pattern the chip uses; focus ring swaps border to `PRIMARY` at 2 px; disabled is a compositing pass (container 12 % / content 38 %, via `StateLayer.disabledContainerOpacity()` / `disabledContentOpacity()`); accessibility role swaps `PUSH_BUTTON` / `TOGGLE_BUTTON` per interaction mode with tooltip / `setName` / `"Icon button"` fallbacks so an icon-only button is never nameless. Keyboard activation on `VK_SPACE` and `VK_ENTER`.
    - **`IconButtonPlaygroundPanels` + standalone `ElwhaIconButtonPlayground`** ([#57](https://github.com/OWS-PFMS/elwha/issues/57)) — new `com.owspfm.elwha.iconbutton.playground` package factoring a 4 × 6 variant gallery (variants × {idle / hover / pressed / selected / focused / disabled}, live `ElwhaIconButton` instances re-skinning end-to-end on mode switch) and a toggle-examples panel (one row per icon-swap pair: `pushPin ↔ pushPinFilled`, `anchor ↔ anchorFilled`, `favorite ↔ favoriteFilled`, `star ↔ starFilled`, each row spans all 4 variants, plus a square-shape demo row at the bottom). The variant gallery's `Hover` and `Pressed` columns pre-render statically via `ElwhaIconButton.setHovered` / `setPressed` so the M3 spec rendering is visible side-by-side without requiring live interaction. `ThemePlayground` gains a top-level `Icon Button` tab between `Chip` and `Surface` composing the same builders.
    - **`IconButtonGroup`** — mutually-exclusive selection across a set of `ElwhaIconButton`s. The toolbar "radio" pattern Swing's `ButtonGroup` provides for `AbstractButton`s; lives here because `ElwhaIconButton` extends `JComponent` directly (the chip / surface / icon-button family all do, to keep paint-time token resolution unencumbered). Non-mandatory mode matches Swing's `ButtonGroup` default (user can click the active button to deselect); mandatory mode mirrors the chip side's `SINGLE_MANDATORY` (deselect refused once a selection exists). Playground's toolbar mockup now demos both patterns side-by-side — independent toggles (pin, anchor) on the left, mandatory radio group (favorite vs star) on the right.
    - **`IconButtonSize` axis + `Sizes` sub-tab + `JToolBar` mockup** ([#59](https://github.com/OWS-PFMS/elwha/issues/59)) — full M3 size axis as `IconButtonSize` enum (`XS(24, 16)` / `S(32, 20)` / `M(40, 24)` / `L(48, 28)` / `XL(56, 32)`; container × icon px), driven by `ElwhaIconButton.setButtonSize(IconButtonSize)`. Defaults to `M`, preserving the prior single-size behavior. Size and shape stay orthogonal — `setButtonSize(L)` leaves the corner radius alone, so the M3-spec shape-tied-to-size coupling is opt-in via `setShape`. The lower-level `setContainerSize(int)` / `setIconSize(int)` setters stay public for off-spec custom sizes (last-write-wins). New `Sizes` sub-tab in the icon-button playground renders a 5 × 4 size × variant matrix plus a real `JToolBar` mockup at `IconButtonSize.S` so the toolbar-usage case is visible end-to-end — answers the smoke-test question by confirming `ElwhaIconButton` works inside `JToolBar` (focus, hover, press, toggle all behave the same as in a free-standing layout). Bundled `favorite_fill` and `star_fill` Material Symbol pairs from the Google Fonts short-term release CDN landed alongside so the toggle-examples panel could drop its state-layer-only rows.
- **ElwhaSurface — token-native rounded surface primitive** ([epic #43](https://github.com/OWS-PFMS/elwha/issues/43)) — a `JPanel` subclass under the new `com.owspfm.elwha.surface` package that paints a role-filled, round-rect, optionally outlined background through the shared `SurfacePainter` helper. Elwha's equivalent of Material's *Paper* — the missing primitive that retires the per-consumer `paintComponent` boilerplate for rounded panels and is the foundation `ElwhaCard` V2 (#253) will compose for its background paint. Decisions: [`docs/research/elwha-surface-design.md`](docs/research/elwha-surface-design.md).
    - **Design doc** ([#51](https://github.com/OWS-PFMS/elwha/issues/51)) — locks the four-setter typed API (`surfaceRole` / `shape` / `borderRole` / `borderWidth`), the `SURFACE` + `MD` + no-border defaults (matched to the locked `ElwhaCard` default so Card V2 inherits the right look), and the explicit out-of-scope list (no elevation, no state layers, no padding API, no raw-typed setters).
    - **`ElwhaSurface` class** ([#52](https://github.com/OWS-PFMS/elwha/issues/52)) — paint-time token resolution via `SurfacePainter` with `overlay = null` (non-interactive); `setOpaque(false)` so the parent isn't pre-painted as a square; fluent setters return `this`. Border is suppressed when `borderRole` is `null`. No interactive state layers — composers needing interactivity wrap Surface inside `ElwhaChip` / the upcoming `ElwhaIconButton` / Card V2.
    - **`SurfacePlaygroundPanels` + standalone `ElwhaSurfacePlayground`** ([#53](https://github.com/OWS-PFMS/elwha/issues/53)) — new `com.owspfm.elwha.surface.playground` package factoring a `ColorRole × ShapeScale` matrix (49 × 7 = 343 cells, each a real `ElwhaSurface`) and a live combo-driven panel. `ThemePlayground` gains a top-level `Surface` tab (sibling of `Chip`) composing the same builders, so the validation matrix and live control surface stay in lockstep across both entry points — same factored-builder pattern `ChipPlaygroundPanels` (#39) established.
- **ElwhaChip token-native rebuild** ([epic #31](https://github.com/OWS-PFMS/elwha/issues/31)) — `ElwhaChip` is the first component rebuilt on the token foundation, and the reference pattern for the components that follow. Rationale and locked decisions: [`docs/research/elwha-flatchip-rebuild.md`](docs/research/elwha-flatchip-rebuild.md). ElwhaChip has zero consumers, so API breakage is accepted; the pre-1.0 no-shims policy applies.
    - **`ChipVariant` is treatment-only** ([#36](https://github.com/OWS-PFMS/elwha/issues/36)) — `FILLED` / `OUTLINED` / `GHOST`, each carrying `surfaceRole()` + `borderRole()` (`ColorRole`). Foreground is never stored on the variant — it is derived as the `on`-pair of the effective surface role.
    - **Typed token setters** ([#37](https://github.com/OWS-PFMS/elwha/issues/37)) — `setSurfaceRole(ColorRole)`, `setShape(ShapeScale)`, `setPadding(SpaceScale horizontal, SpaceScale vertical)`. Cohesion guarantee: callers pick a role / scale step, never a raw `Color` / `Insets` / pixel value.
    - **Foreground = `on`-pair of the effective surface role** — V1's luma / auto-contrast machinery (`AUTO_FG_DARK` / `AUTO_FG_LIGHT`, `isLight`, `effectiveSurfaceForContrast`) deleted entirely. Always correct by construction.
    - **Uniform state-layer overlays** — hover / pressed / selected feedback composited through the locked `StateLayer` model (8% / 10% / 12%), tinted by the surface's `on`-role. V1's per-variant `selectedFillStrength` is gone. Selected swaps the border to `PRIMARY` so OUTLINED chips read as "the picked one" even under the uniform 12% selected overlay (rebuild doc §9 Q2).
    - **Default shape `SM` (8 px)** — replaces V1's capsule (`999`). `setShape(ShapeScale.FULL)` restores the pill look.
    - **M3 chip-type factory presets** ([#38](https://github.com/OWS-PFMS/elwha/issues/38)) — `ElwhaChip.assistChip(text)`, `filterChip(text)`, `inputChip(text, onRemove)`, `suggestionChip(text)`. Sugar over the orthogonal axes; everything stays overridable.
    - **Internal `SurfacePainter` helper** — extracted at `com.owspfm.elwha.theme.SurfacePainter`. Owns the round-rect fill + border + state-layer overlay so the round-rect surface paint lives in exactly one place across the component set — `ElwhaChip` consumes it today; `ElwhaSurface` ([#43](https://github.com/OWS-PFMS/elwha/issues/43)) and `ElwhaCard` V2 will consume it when they land. Library-internal — declared `public` only to cross package boundaries, not part of the public API.
    - **`ChipPlaygroundPanels`** ([#39](https://github.com/OWS-PFMS/elwha/issues/39)) — new `com.owspfm.elwha.chip.playground` package factoring the variant-gallery + live-list builders out of the standalone playground so they can be reused. Standalone `ElwhaChipPlayground` and `ThemePlayground`'s new `Chip` tab both compose the same builders.
- **Design-token foundation** ([epic #30](https://github.com/OWS-PFMS/elwha/issues/30)) — the token layer the design-system direction is built on, implemented under the new `com.owspfm.elwha.theme` package. No component code; foundation only.
    - **Token facade enums** ([#32](https://github.com/OWS-PFMS/elwha/issues/32)) — `ColorRole` (49 roles, the full M3 scheme, with `on`-pairing lookup and baseline fallback), `ShapeScale` (7), `SpaceScale` (6), `TypeRole` (12), `StateLayer` (5, with sRGB overlay blending). Every enum resolves from `UIManager` under the `Elwha.*` namespace at call time per the taxonomy's paint-time binding rule.
    - **`ElwhaTheme` install API** ([#33](https://github.com/OWS-PFMS/elwha/issues/33)) — `Palette` (completeness-validated 49-role color map), `Theme` (`{light, dark}`), `Mode` (`LIGHT` / `DARK` / `SYSTEM` with install-time OS detection), `Typography`, `Config`, and the static `ElwhaTheme.install(Config)` facade running the locked 8-step sequence. Idempotent and re-callable — runtime theme/mode switching is just re-installing. Every color and font is stored as a `ColorUIResource` / `FontUIResource` so `updateComponentTreeUI` correctly re-skins live components on a mode switch.
    - **FlatLaf-native key mapping** ([#34](https://github.com/OWS-PFMS/elwha/issues/34)) — `FlatLafKeyMapping` maps a curated ~120-key subset of FlatLaf's `UIManager` keys onto token roles, and bakes M3 hover / pressed state-layer overlays into FlatLaf's discrete `*hoverBackground` / `*pressedBackground` keys so raw Swing inherits the design language. Enumerated as Appendix A of `elwha-theme-install-api.md`. The locked **focus model** is a `PRIMARY`-colored 2 px ring (`Component.focusWidth = 2`, `Component.focusColor = PRIMARY`); every `*.focused*` and `*.hover*` key is mapped explicitly to its resting equivalent — an unmapped focus key silently falls back to FlatLaf's blue accent, so the indicator is the ring alone and the resting fill / border is preserved. `Slider.trackValueColor` (the filled portion) is mapped to `PRIMARY` so the slider's active track is on-palette.
    - **Baseline palette + JSON loader** ([#35](https://github.com/OWS-PFMS/elwha/issues/35)) — `MaterialPalettes.baseline()` ships the M3 baseline scheme as a Elwha-normalized JSON resource loaded by `PaletteLoader`. Schema and the M3-export conversion are Appendix B of `elwha-theme-install-api.md`. Four roles the current M3 builder omits (`surfaceTint`, `background`, `onBackground`, `surfaceVariant`) are derived by canonical M3 rule.
- **Checkbox + radio split via FlatLaf icon style** — `RadioButton.icon.style = "outlined"` separates the shared `CheckBox.icon.*` palette so the checkbox keeps the filled M3 look (primary fill, white check) while the radio renders as the canonical M3 ring (primary 2 px ring, transparent interior, primary center dot when checked). `CheckBox.icon.focusedSelectedBackground` is mapped explicitly because its fallback is `focusedBackground` (not `selectedBackground`), and the focus-ring model would otherwise make a focused, checked icon render as empty.
- **Bundled Inter font** — the Inter Regular and Medium faces ship under `src/main/resources/com/owspfm/elwha/theme/fonts/` (SIL OFL 1.1), giving `Typography.defaults()` real M3 400/500 weight rendering. Attribution added to `NOTICE`.
- **`MaterialIcons` sized overloads** — every lookup (`pushPin`, `delete`, …) now has a sized companion (`pushPin(int size)`, etc.) so consumers don't have to chain `.derive(size, size)`.
- **`MaterialIcons.pair(name)` outline / fill helper** — convenience for the canonical M3 toggle-icon-swap pattern. Returns a `MaterialIcons.IconPair` record with `resting()` + `filled()` accessors derived by appending `_fill` to the base name (e.g. `pair("push_pin")` loads `push_pin.svg` + `push_pin_fill.svg`). Lets `button.setIcons(p.resting(), p.filled())` be a one-liner without contaminating `ElwhaIconButton` with icon-library knowledge — the outline/fill convention lives in the icon library that actually knows Material Symbol naming, not in the button. Sized companion `pair(name, size)` too.
- **`ThemePlayground`** — interactive visual harness for the token foundation and the components built on it. Four top-level tabs: `Color Roles` (all 49 swatches with luminance-picked label color for non-pairing roles), `Type Scale` (the 12 type roles rendered with bundled Inter), `Swing Comps` (raw Swing widgets inheriting the theme through the FlatLaf-native key mapping, with six vertically-stacked sub-rows including a borderless M3 fill-toggle icon button and a segmented icon-toggle group), and `Chip` (an inner tabbed pane with the chip variant gallery and the live list, reused from `ChipPlaygroundPanels`). A live light / dark / system mode toggle exercises the install API's runtime-switch path.

### Changed

- **Library renamed `FlatComp` → `Elwha`** ([#42](https://github.com/OWS-PFMS/elwha/issues/42)) — one atomic sweep removing the substrate-anchored "Flat" name and flattening the package path. Decision rationale: [`docs/research/elwha-rename-decision.md`](docs/research/elwha-rename-decision.md). **Breaking** at every public surface; the lib has zero theme-API consumers and the one component consumer (OWS-Local-Search-GUI) is dev-paused, so no compat shims are provided per the pre-1.0 policy.
    - **Artifact:** `com.owspfm:flatcomp` → `com.owspfm:elwha`; GitHub repo `OWS-PFMS/flatcomp` → `OWS-PFMS/elwha` (post-merge admin step).
    - **Packages:** `com.owspfm.ui.components.<x>` → `com.owspfm.elwha.<x>` for `theme` / `card` / `chip` / `icons` (incl. sub-packages); `com.owspfm.ui.components.flatlist` → `com.owspfm.elwha.list`. Resource paths under `src/main/resources/com/owspfm/ui/components/` move in lockstep (`theme/palettes`, `theme/fonts`, `card/list/cursors`). The icon resource path `com/owspfm/icons/material/` is unaffected.
    - **Classes:** `FlatCard` → `ElwhaCard`, `FlatChip` → `ElwhaChip`, `FlatList` → `ElwhaList`, `FlatListOrientation` → `ElwhaListOrientation`, plus the list containers (`FlatCardList` → `ElwhaCardList`, `FlatChipList` → `ElwhaChipList`), playgrounds (`FlatCardPlayground` → `ElwhaCardPlayground`, `FlatChipPlayground` → `ElwhaChipPlayground`, `FlatCardListShowcase` → `ElwhaCardListShowcase`), demos (`FlatCardDemo`/`FlatChipDemo`/`FlatCardListDemo` → `Elwha*`), and the theme entry point `FlatCompTheme` → `ElwhaTheme`. `FlatLafKeyMapping` is unchanged — its prefix refers to FlatLaf, not the library.
    - **UIManager namespace:** all `FlatComp.*` keys renamed to `Elwha.*` (`Elwha.color.*`, `Elwha.shape.*`, `Elwha.space.*`, `Elwha.type.*`, `Elwha.state.*`).
    - **Locked-doc amendment:** [`elwha-token-taxonomy.md`](docs/research/elwha-token-taxonomy.md) and [`elwha-theme-install-api.md`](docs/research/elwha-theme-install-api.md) received a mechanical namespace amendment only (`FlatComp.*` → `Elwha.*`, `FlatCompTheme` → `ElwhaTheme`). The locked decisions are not reopened.
- **`MaterialIcons.DEFAULT_SIZE`: 14 → 24** (M3 icon-button standard). The previous 14 px was chip-specialized; Material Symbols are designed at the 20-dp optical-size axis, so 24 px keeps them at their design-intended visual weight for icon buttons and toolbars. **Breaking visible change** for any consumer calling the no-arg `MaterialIcons.foo()` form — icons will render larger by default. Internally, the 13 chip-context call sites in `ElwhaChipList` and `ElwhaChipPlayground` are pinned to an explicit `(14)` so chip visuals are preserved; chip-icon sizing will be revisited during the ElwhaChip V2 refresh.
- **`FlatPill` renamed to `ElwhaChip`** ([epic #27](https://github.com/OWS-PFMS/elwha/issues/27)) — aligns the component name with Material Design's chip taxonomy. **Breaking change** for any consumer importing the old names, even though no consumer has shipped yet against this lib. The rename is mechanical and 1:1; no behavior changes.
    - **Package:** `com.owspfm.ui.components.pill` → `com.owspfm.elwha.chip`; `pill.list` → `chip.list`.
    - **Classes:** every `FlatPill*` and `Pill*` type renamed (`FlatPill` → `ElwhaChip`, `FlatPillList` → `ElwhaChipList`, `FlatPillPlayground` → `ElwhaChipPlayground`, `FlatPillDemo` → `ElwhaChipDemo`, `PillVariant` → `ChipVariant`, `PillInteractionMode` → `ChipInteractionMode`, `PillListModel` / `DefaultPillListModel` → `ChipListModel` / `DefaultChipListModel`, `PillSelectionModel` / `DefaultPillSelectionModel` → `ChipSelectionModel` / `DefaultChipSelectionModel`, `PillSelectionMode` → `ChipSelectionMode`, `PillSelectionEvent` / `PillSelectionListener` → `ChipSelectionEvent` / `ChipSelectionListener`, `PillAdapter` → `ChipAdapter`, `PillReorderEvent` / `PillReorderListener` → `ChipReorderEvent` / `ChipReorderListener`, `PillListDataEvent` / `PillListDataListener` → `ChipListDataEvent` / `ChipListDataListener`).
    - **UIManager keys:** every `"FlatPill.*"` string value renamed to `"ElwhaChip.*"` (`ElwhaChip.background`, `ElwhaChip.hoverBackground`, `ElwhaChip.padding`, `ElwhaChip.arc`, `ElwhaChip.warmAccent`, `ElwhaChip.foreground`, …). Java constant names (`K_*`, `STYLE_PROPERTY`) are unchanged; only the string values move. The client-property key `"FlatPill.style"` is now `"ElwhaChip.style"`, and the ad-hoc `"FlatPill.removeIcon"` override slot used by the playground is now `"ElwhaChip.removeIcon"`.
    - **Consumer migration:** any consumer that previously overrode `UIManager.put("FlatPill.*", ...)` at app startup must update its overrides to the `ElwhaChip.*` namespace. Imports of `com.owspfm.ui.components.pill.*` must update to `com.owspfm.elwha.chip.*`. Tracked downstream in [OWS-PFMS/OWS-Local-Search-GUI#258](https://github.com/OWS-PFMS/OWS-Local-Search-GUI/issues/258).
    - **History:** the component was originally named `FlatPill` because its shape is a capsule ("pill"); the Material industry-standard name is "chip" for interactive token elements (input / choice / filter / action chips). Pre-flight inventory of every renamed touchpoint lives at `docs/research/flatpill-to-flatchip-rename-inventory.md`.
- **Naming convention switched to Google Java Style** ([epic #5](https://github.com/OWS-PFMS/elwha/issues/5)). Every `my*`-prefixed instance field and every `the*`-prefixed parameter / local was renamed to its unprefixed equivalent across the source tree. Public API parameter names move with this change — Java does not bind to parameter names at the call site, but Javadoc `@param` references and consumers reading reflection metadata will see the new names. No method signatures, class names, package names, or behavior changed.
- **`ChipVariant.GHOST` does not participate in selection rendering** ([#50](https://github.com/OWS-PFMS/elwha/issues/50), rebuild doc §9 Q2 amendment) — GHOST is Elwha's equivalent of M3's Text button (the lowest of M3's five emphasis levels), and M3 doesn't define a selected state at that emphasis level. Resolves a visual collapse surfaced during PR #46 smoketest: under the preemptive `PRIMARY` border swap on selected, `GHOST + SELECTED` and `OUTLINED + SELECTED` rendered identically (both: no fill + 12 % overlay + 1 px `PRIMARY` border), and `GHOST + SELECTED` additionally read as a faded `FILLED` chip — defeating GHOST's "no surface unless interacting" intent. The `selected` field and `PROPERTY_SELECTED` events are preserved on the API surface (so `setVariant(...)` later to a non-GHOST variant resumes rendering the selected state), but the rendering pipeline ignores `selected` while variant is `GHOST` — no `SELECTED` state-layer overlay, no border swap. Consumers needing a togglable chip with a visible selected state should use `OUTLINED` (M3's default treatment for filter chips). `ChipVariant × ChipInteractionMode` orthogonality is preserved at the API level — only the visual contract changes.

### Removed

- **`ChipVariant.WARM_ACCENT`** (epic #31) — never an M3 concept and bundled a color choice into what should be a treatment axis. The secondary-container look is now `FILLED` + `setSurfaceRole(SECONDARY_CONTAINER)` — color and treatment cleanly orthogonal.
- **`ElwhaChip.*` UIManager namespace** (epic #31) — the entire ~12-key namespace (`ElwhaChip.background`, `borderColor`, `arc`, `padding`, `hoverBackground`, `pressedBackground`, `selectedBackground`, `selectedBorderColor`, `focusColor`, `disabledBackground`, `warmAccent`, `foreground`) and the `K_*` public constants are deleted. App-wide chip theming now happens through the installed palette + the variant's role declarations; per-instance happens through the typed setters. The `"ElwhaChip.style"` client-property escape hatch is also gone — typed setters cover what it did.
- **`ElwhaChip` raw-typed setters** (epic #31) — `setSurfaceColor(Color)`, `setForegroundColor(Color)`, `setCornerRadius(Integer)`, `setBorderColor(Color)`, `setPadding(Insets)`, `setPadding(int, int)`. Replaced by `setSurfaceRole(ColorRole)`, `setShape(ShapeScale)`, `setPadding(SpaceScale, SpaceScale)`. `setBorderWidth(int)` is **kept** — stroke width is genuine geometry with no token equivalent.
- **`ElwhaChip` foreground override** (epic #31) — no per-instance foreground setter. Foreground is always the `on`-pair of the effective surface role, so an independent override would re-introduce unpaired surface/foreground.
- **`ElwhaChip` auto-contrast machinery** (epic #31) — `resolveForegroundColor()`'s luma branch, `AUTO_FG_DARK` / `AUTO_FG_LIGHT`, `isLight`, `effectiveSurfaceForContrast`. The token system's `on`-pairing makes them unnecessary.
- **`ElwhaChipDemo`** (epic #31) — the minimal smoke-test demo. Redundant with `ElwhaChipPlayground` after the playground rework; the playground now covers the same matrix plus the live-list surface.

### Fixed

- **`SurfacePainter` stroke geometry for borders wider than 1 px** (surfaced by smoketesting [#43](https://github.com/OWS-PFMS/elwha/issues/43)) — the painter centered the stroke on a path inset by `0.5 px` (correct for a 1 px AA line on the half-pixel grid), so any wider stroke extended outside the component bounds and got clipped: at width 8 only ~4.5 px of fuzzy inner stroke was visible instead of a crisp outer ring. Fixed by insetting the stroke centerline by `borderWidth / 2` and shrinking the corner diameter by `borderWidth` so the stroke's outer edge tracks the fill's corner exactly. Fill now extends to the full component bounds for clean AA at the corners. Latent under `ElwhaChip` (max stroke 2 px, artifact below the visibility threshold); newly user-visible via `ElwhaSurface.setBorderWidth(int)`.
- **`ElwhaChip` text baseline drift when leading affordance is hidden** ([#49](https://github.com/OWS-PFMS/elwha/issues/49)) — chips with no visible leading affordance but a visible trailing icon read with the text sitting ~2px high relative to the trailing icon's centerline. Root cause: stock `FlowLayout` skipped invisible children for row-height computation, so the leading cluster reported a shorter preferred height than the trailing button's hit-target; when `BorderLayout` stretched the cluster to match, the slack landed below the row instead of being split above and below. Fixed by an internal `BaselineCenteringFlowLayout` that floors the row height at `BUTTON_MIN_HIT_TARGET` and re-centers the row after the parent's layout pass. Predates epic #31; surfaced by smoketesting the chip rebuild.
- **Bundled `MaterialPalettes.baseline()` regenerated from a fresh Material Theme Builder export** ([#47](https://github.com/OWS-PFMS/elwha/issues/47)) — the prior baseline scheme produced muddy / low-legibility chip surfaces, and additionally collapsed the secondary role onto primary (`schemes.light.primary == schemes.light.secondary`), nullifying the primary/secondary emphasis hierarchy M3 components are designed around. New palette is a clean single-seed M3 scheme (`#6750A4`); secondary and tertiary tonal palettes are derived by canonical hue-rotation, so `primary` / `secondary` / `tertiary` (and their containers) are distinct in both light and dark. All 49 `ColorRole` keys now come directly from `schemes.light` / `schemes.dark` (no derived roles). Raw export archived at `docs/research/baseline-palette-mtb-export-2026-05-15.json`.

### Tooling

- **Spotless** (`spotless-maven-plugin` 2.46.1 + `googleJavaFormat`) wired into the Maven build as `mvn spotless:check` at the `verify` phase. Run `mvn spotless:apply` locally to fix formatting. Note: requires JDK 21 — Spotless + google-java-format hits a binary-API drift on JDK 22+ (tracked upstream at [diffplug/spotless#2468](https://github.com/diffplug/spotless/issues/2468)).
- **Checkstyle** (`maven-checkstyle-plugin` 3.6.0 + Checkstyle 10.21.0) wired with a project-local config (`config/checkstyle/checkstyle.xml`) that enforces Google naming patterns plus a project-specific ban on `my*` / `the*` identifier prefixes. ## [0.1.0] — 2026-05-12

Initial release. Library extracted from the [OWS-Local-Search-GUI](https://github.com/OWS-PFMS/OWS-Local-Search-GUI) project ([epic #231](https://github.com/OWS-PFMS/OWS-Local-Search-GUI/issues/231)).

### Added

- **`ElwhaCard`** — theme-aware card primitive with header, body, surface variants, hover/pressed/selected states, optional collapse/expand, leading icon, trailing actions.
- **`ElwhaCardList<T>`** — list of `ElwhaCard` items: selection, drag-to-reorder, filter, sort, orientation modes (vertical / horizontal / wrap / grid).
- **`FlatPill`** — compact pill primitive: text + optional leading icon + optional trailing action button, themeable via UIManager keys (`FlatPill.*`), auto-contrast foreground via BT.601 luma.
- **`FlatPillList<T>`** — list of `FlatPill` items with selection modes (`NONE` / `SINGLE` / `SINGLE_MANDATORY` / `MULTIPLE`), drag-to-reorder, pinned-partition + anchored modes (`MovementMode` enum), icon affordances (`IconAffordance` enum).
- **`ElwhaList<T>`** — shared cross-cutting interface implemented by both `ElwhaCardList` and `FlatPillList` (orientation, gap, padding, empty / loading state, filter, sort).
- **`MaterialIcons`** — helper that exposes 17 Material Symbols SVGs (Rounded / 400 / fill0 / 20px) via `FlatSVGIcon` with a theme-aware color filter.
- **`ElwhaCardPlayground`** — interactive playground for the card family (embeds `ElwhaCardListShowcase` + `GalleryPanel` panels for the list view, single-card variants, and cycle/cursor examples).
- **`FlatPillPlayground`** — interactive playground for the pill list family, including a LAF tweak panel for live UIManager-key experimentation.

### Notes

- API is **not yet stable** at 0.1.0. Breaking changes between minor versions are expected until 1.0.0.
- Two breaking-change refactors are tracked and gated on this release:
  - `FlatPill` → `ElwhaChip` rename (taxonomy alignment with Material)
  - `ElwhaList<T>` extension to share selection + drag-reorder surface across both list families
- 1.0.0 will follow those landings.
