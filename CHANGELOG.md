## [Unreleased]

### Added

- **ElwhaButton ‚Äî token-native M3 text-button primitive** ([epic #103](https://github.com/OWS-PFMS/elwha/issues/103)) ‚Äî a rounded-rect text-button primitive (planned for the new `com.owspfm.elwha.button` package) with five M3 Expressive variants (`ELEVATED` / `FILLED` / `FILLED_TONAL` / `OUTLINED` / `TEXT`), a five-size axis (`XS` / `S` / `M` / `L` / `XL`), round + square shape options, and a `CLICKABLE` / `SELECTABLE` interaction axis. Completes the action-row vocabulary alongside `ElwhaIconButton` (epic #45) ‚Äî V3 Card spec ¬ß3.3 pairs Outlined cards with Filled-Tonal buttons, which has no path through the lib today; ElwhaButton closes that gap and blocks the v0.2.0 release. Decisions and the deliberate M3 divergences: [`docs/research/elwha-button-design.md`](docs/research/elwha-button-design.md).
    - **Design doc** ([epic #103](https://github.com/OWS-PFMS/elwha/issues/103)) ‚Äî locks the variant table, the **M3 Expressive baseline posture** (adopt sizes XS/M/L/XL, square shape, 16dp Small padding, and toggle interaction without flagging as divergence), the **hybrid selection color model** (variant-specific surface swap for `FILLED` + `OUTLINED` per M3 spec, uniform 12% `SELECTED` overlay + `PRIMARY` border swap for `ELEVATED` + `FILLED_TONAL`, `IllegalStateException` thrown on `TEXT + SELECTABLE`), the **paint-helper plan** (new `RipplePainter` extracted from `ElwhaCard.paintRipple`, new `ShadowPainter` extracted from `SurfacePainter.renderShadowImage` with a redesigned `(arc, elevation)` cache key + 9-slice rendering that kills PR #110's `setSuspendShadowRecompute` workaround at the root), per-size measurements and the in-component `BUTTON_SQUARE_CORNERS_DP` lookup (12 / 12 / 16 / 28 / 28 dp), 48 dp a11y target inflation for XS / S sizes, and ¬ß10's explicit table of every place Elwha intentionally diverges from both base M3 and M3 Expressive (shape morph deferred, focus indicator via border-swap for `ElwhaIconButton` parity, ripple clipped to container, single Gaussian shadow vs M3's key+ambient pair). Includes [`docs/research/shadow-spike-2026-05-19.md`](docs/research/shadow-spike-2026-05-19.md): the spike that disqualified `FlatLaf.FlatDropShadowBorder` on visual grounds (rectangular halo doesn't follow round corners) but adopted its size-independent caching insight (`(arc, elevation)`-keyed shared cache + 9-slice paint).
    - **Paint helpers ‚Äî `RipplePainter` + `ShadowPainter`** ([#115](https://github.com/OWS-PFMS/elwha/issues/115)) ‚Äî two new static helpers under `com.owspfm.elwha.theme`, parallel to `SurfacePainter`. `RipplePainter` is the M3-style expanding-circle press ripple (250 ms expand + 150 ms fade) ported verbatim from `ElwhaCard.paintRipple`; the painter is stateless and does not own the animation timer (caller seeds `progress`). `ShadowPainter` replaces the per-instance `(bodyW, bodyH, arc, elevation)` cache in `ElwhaSurface` with a shared `(arc, elevation)`-keyed `SoftReference` cache and 9-slice rendering, so shadow cost is decoupled from body size ‚Äî the foundation for follow-up F2 (#122) which migrates `ElwhaSurface` onto the new helper and deletes PR #110's `setSuspendShadowRecompute` workaround. `RipplePainterDemo` and `ShadowPainterDemo` under `com.owspfm.elwha.theme.playground` are visual smoke-tests; `ShadowPainterDemo` includes a microbench panel reporting cached-9-slice-paint vs cold-`ConvolveOp`-render speedup.
- **`MaterialIcons` expand/overflow glyphs** ([#81](https://github.com/OWS-PFMS/elwha/issues/81)) ‚Äî three Material Symbols Rounded / 400 / fill 0 / 20px SVGs (`expand_more`, `expand_less`, `more_vert`) under `src/main/resources/com/owspfm/icons/material/` plus the matching `MaterialIcons.expandMore()` / `expandLess()` / `moreVert()` accessors (each with a sized overload). Salvaged from the abandoned V2 PR stack ahead of the V3 build (epic [#80](https://github.com/OWS-PFMS/elwha/issues/80)) ‚Äî these glyphs are load-bearing for V3's chevron and overflow-trigger affordances.
- **Cross-component API consistency sweep** ([#62](https://github.com/OWS-PFMS/elwha/issues/62)) ‚Äî locks the doctrine new components are expected to match. Doc: [`docs/development/component-api-conventions.md`](docs/development/component-api-conventions.md), referenced from `CLAUDE.md` Conventions. Four rules: bare `getX()` returns the effective value (no `getEffectiveX()` parallel); every variant-bearing component exposes one `static` factory per variant taking the convenience-constructor argument; single-arg convenience constructors only (no proliferating overloads); variant-bearing components do not expose `setBorderRole(ColorRole)` ‚Äî border role is variant-derived. The doctrine subsumes the earlier "variant-as-behavior vs variant-as-treatment" framing.
    - **`ElwhaIconButton.filledIconButton(Icon)` / `filledTonalIconButton(Icon)` / `outlinedIconButton(Icon)` / `standardIconButton(Icon)`** ‚Äî four M3 emphasis-level factory presets matching `ElwhaChip`'s pattern. Each delegates to `new ElwhaIconButton(icon).setVariant(...)`; every preset choice remains overridable through the normal setters.
    - **`ElwhaChip.getBorderWidth()`** ‚Äî companion getter to `setBorderWidth(int)`. Asymmetric setter-without-getter was drift; brings `ElwhaChip` in line with `ElwhaSurface` and `ElwhaIconButton`.
- **ElwhaIconButton ‚Äî token-native M3 icon-button primitive** ([epic #45](https://github.com/OWS-PFMS/elwha/issues/45)) ‚Äî a square, icon-only button under the new `com.owspfm.elwha.iconbutton` package with four M3 emphasis variants (`FILLED` / `FILLED_TONAL` / `OUTLINED` / `STANDARD`), a `CLICKABLE` / `SELECTABLE` interaction axis, and a declarative `setIcons(resting, selected)` toggle pattern. The third token-native component built on `SurfacePainter`, after `ElwhaChip` (#31) and `ElwhaSurface` (#43); subsumes the playground's hand-rolled `pushPin ‚Üî pushPinFilled` borderless-toggle pattern. Decisions and the deliberate M3 divergences (uniform state-layer overlay vs M3's per-variant surface swap, single 40 dp size, no shape morphing): [`docs/research/elwha-icon-button-design.md`](docs/research/elwha-icon-button-design.md).
    - **Design doc** ([#55](https://github.com/OWS-PFMS/elwha/issues/55)) ‚Äî locks the variant table (`FILLED` ‚Üí `PRIMARY`, `FILLED_TONAL` ‚Üí `SECONDARY_CONTAINER`, `OUTLINED` ‚Üí transparent + `OUTLINE`, `STANDARD` ‚Üí transparent + no border), the foreground-resolution rule (`on`-pair of the effective surface, falling back to `ON_SURFACE_VARIANT`, with the single `STANDARD + selected` ‚Üí `PRIMARY` exception), the defaults (STANDARD + CLICKABLE + `ShapeScale.FULL` + 40 dp container + 24 dp icon), and ¬ß10's explicit table of every place Elwha intentionally diverges from the M3 icon-button spec.
    - **`ElwhaIconButton` class + `IconButtonVariant` + `IconButtonInteractionMode`** ([#56](https://github.com/OWS-PFMS/elwha/issues/56)) ‚Äî fluent setters return `this`; paint delegates to `SurfacePainter` (no round-rect code in this class); icon tinting via the same `FlatSVGIcon.ColorFilter` pattern the chip uses; focus ring swaps border to `PRIMARY` at 2 px; disabled is a compositing pass (container 12 % / content 38 %, via `StateLayer.disabledContainerOpacity()` / `disabledContentOpacity()`); accessibility role swaps `PUSH_BUTTON` / `TOGGLE_BUTTON` per interaction mode with tooltip / `setName` / `"Icon button"` fallbacks so an icon-only button is never nameless. Keyboard activation on `VK_SPACE` and `VK_ENTER`.
    - **`IconButtonPlaygroundPanels` + standalone `ElwhaIconButtonPlayground`** ([#57](https://github.com/OWS-PFMS/elwha/issues/57)) ‚Äî new `com.owspfm.elwha.iconbutton.playground` package factoring a 4 √ó 6 variant gallery (variants √ó {idle / hover / pressed / selected / focused / disabled}, live `ElwhaIconButton` instances re-skinning end-to-end on mode switch) and a toggle-examples panel (one row per icon-swap pair: `pushPin ‚Üî pushPinFilled`, `anchor ‚Üî anchorFilled`, `favorite ‚Üî favoriteFilled`, `star ‚Üî starFilled`, each row spans all 4 variants, plus a square-shape demo row at the bottom). The variant gallery's `Hover` and `Pressed` columns pre-render statically via `ElwhaIconButton.setHovered` / `setPressed` so the M3 spec rendering is visible side-by-side without requiring live interaction. `ThemePlayground` gains a top-level `Icon Button` tab between `Chip` and `Surface` composing the same builders.
    - **`IconButtonGroup`** ‚Äî mutually-exclusive selection across a set of `ElwhaIconButton`s. The toolbar "radio" pattern Swing's `ButtonGroup` provides for `AbstractButton`s; lives here because `ElwhaIconButton` extends `JComponent` directly (the chip / surface / icon-button family all do, to keep paint-time token resolution unencumbered). Non-mandatory mode matches Swing's `ButtonGroup` default (user can click the active button to deselect); mandatory mode mirrors the chip side's `SINGLE_MANDATORY` (deselect refused once a selection exists). Playground's toolbar mockup now demos both patterns side-by-side ‚Äî independent toggles (pin, anchor) on the left, mandatory radio group (favorite vs star) on the right.
    - **`IconButtonSize` axis + `Sizes` sub-tab + `JToolBar` mockup** ([#59](https://github.com/OWS-PFMS/elwha/issues/59)) ‚Äî full M3 size axis as `IconButtonSize` enum (`XS(24, 16)` / `S(32, 20)` / `M(40, 24)` / `L(48, 28)` / `XL(56, 32)`; container √ó icon px), driven by `ElwhaIconButton.setButtonSize(IconButtonSize)`. Defaults to `M`, preserving the prior single-size behavior. Size and shape stay orthogonal ‚Äî `setButtonSize(L)` leaves the corner radius alone, so the M3-spec shape-tied-to-size coupling is opt-in via `setShape`. The lower-level `setContainerSize(int)` / `setIconSize(int)` setters stay public for off-spec custom sizes (last-write-wins). New `Sizes` sub-tab in the icon-button playground renders a 5 √ó 4 size √ó variant matrix plus a real `JToolBar` mockup at `IconButtonSize.S` so the toolbar-usage case is visible end-to-end ‚Äî answers the smoke-test question by confirming `ElwhaIconButton` works inside `JToolBar` (focus, hover, press, toggle all behave the same as in a free-standing layout). Bundled `favorite_fill` and `star_fill` Material Symbol pairs from the Google Fonts short-term release CDN landed alongside so the toggle-examples panel could drop its state-layer-only rows.
- **ElwhaSurface ‚Äî token-native rounded surface primitive** ([epic #43](https://github.com/OWS-PFMS/elwha/issues/43)) ‚Äî a `JPanel` subclass under the new `com.owspfm.elwha.surface` package that paints a role-filled, round-rect, optionally outlined background through the shared `SurfacePainter` helper. Elwha's equivalent of Material's *Paper* ‚Äî the missing primitive that retires the per-consumer `paintComponent` boilerplate for rounded panels and is the foundation `ElwhaCard` V2 (#253) will compose for its background paint. Decisions: [`docs/research/elwha-surface-design.md`](docs/research/elwha-surface-design.md).
    - **Design doc** ([#51](https://github.com/OWS-PFMS/elwha/issues/51)) ‚Äî locks the four-setter typed API (`surfaceRole` / `shape` / `borderRole` / `borderWidth`), the `SURFACE` + `MD` + no-border defaults (matched to the locked `ElwhaCard` default so Card V2 inherits the right look), and the explicit out-of-scope list (no elevation, no state layers, no padding API, no raw-typed setters).
    - **`ElwhaSurface` class** ([#52](https://github.com/OWS-PFMS/elwha/issues/52)) ‚Äî paint-time token resolution via `SurfacePainter` with `overlay = null` (non-interactive); `setOpaque(false)` so the parent isn't pre-painted as a square; fluent setters return `this`. Border is suppressed when `borderRole` is `null`. No interactive state layers ‚Äî composers needing interactivity wrap Surface inside `ElwhaChip` / the upcoming `ElwhaIconButton` / Card V2.
    - **`SurfacePlaygroundPanels` + standalone `ElwhaSurfacePlayground`** ([#53](https://github.com/OWS-PFMS/elwha/issues/53)) ‚Äî new `com.owspfm.elwha.surface.playground` package factoring a `ColorRole √ó ShapeScale` matrix (49 √ó 7 = 343 cells, each a real `ElwhaSurface`) and a live combo-driven panel. `ThemePlayground` gains a top-level `Surface` tab (sibling of `Chip`) composing the same builders, so the validation matrix and live control surface stay in lockstep across both entry points ‚Äî same factored-builder pattern `ChipPlaygroundPanels` (#39) established.
- **ElwhaChip token-native rebuild** ([epic #31](https://github.com/OWS-PFMS/elwha/issues/31)) ‚Äî `ElwhaChip` is the first component rebuilt on the token foundation, and the reference pattern for the components that follow. Rationale and locked decisions: [`docs/research/elwha-flatchip-rebuild.md`](docs/research/elwha-flatchip-rebuild.md). ElwhaChip has zero consumers, so API breakage is accepted; the pre-1.0 no-shims policy applies.
    - **`ChipVariant` is treatment-only** ([#36](https://github.com/OWS-PFMS/elwha/issues/36)) ‚Äî `FILLED` / `OUTLINED` / `GHOST`, each carrying `surfaceRole()` + `borderRole()` (`ColorRole`). Foreground is never stored on the variant ‚Äî it is derived as the `on`-pair of the effective surface role.
    - **Typed token setters** ([#37](https://github.com/OWS-PFMS/elwha/issues/37)) ‚Äî `setSurfaceRole(ColorRole)`, `setShape(ShapeScale)`, `setPadding(SpaceScale horizontal, SpaceScale vertical)`. Cohesion guarantee: callers pick a role / scale step, never a raw `Color` / `Insets` / pixel value.
    - **Foreground = `on`-pair of the effective surface role** ‚Äî V1's luma / auto-contrast machinery (`AUTO_FG_DARK` / `AUTO_FG_LIGHT`, `isLight`, `effectiveSurfaceForContrast`) deleted entirely. Always correct by construction.
    - **Uniform state-layer overlays** ‚Äî hover / pressed / selected feedback composited through the locked `StateLayer` model (8% / 10% / 12%), tinted by the surface's `on`-role. V1's per-variant `selectedFillStrength` is gone. Selected swaps the border to `PRIMARY` so OUTLINED chips read as "the picked one" even under the uniform 12% selected overlay (rebuild doc ¬ß9 Q2).
    - **Default shape `SM` (8 px)** ‚Äî replaces V1's capsule (`999`). `setShape(ShapeScale.FULL)` restores the pill look.
    - **M3 chip-type factory presets** ([#38](https://github.com/OWS-PFMS/elwha/issues/38)) ‚Äî `ElwhaChip.assistChip(text)`, `filterChip(text)`, `inputChip(text, onRemove)`, `suggestionChip(text)`. Sugar over the orthogonal axes; everything stays overridable.
    - **Internal `SurfacePainter` helper** ‚Äî extracted at `com.owspfm.elwha.theme.SurfacePainter`. Owns the round-rect fill + border + state-layer overlay so the round-rect surface paint lives in exactly one place across the component set ‚Äî `ElwhaChip` consumes it today; `ElwhaSurface` ([#43](https://github.com/OWS-PFMS/elwha/issues/43)) and `ElwhaCard` V2 will consume it when they land. Library-internal ‚Äî declared `public` only to cross package boundaries, not part of the public API.
    - **`ChipPlaygroundPanels`** ([#39](https://github.com/OWS-PFMS/elwha/issues/39)) ‚Äî new `com.owspfm.elwha.chip.playground` package factoring the variant-gallery + live-list builders out of the standalone playground so they can be reused. Standalone `ElwhaChipPlayground` and `ThemePlayground`'s new `Chip` tab both compose the same builders.
- **Design-token foundation** ([epic #30](https://github.com/OWS-PFMS/elwha/issues/30)) ‚Äî the token layer the design-system direction is built on, implemented under the new `com.owspfm.elwha.theme` package. No component code; foundation only.
    - **Token facade enums** ([#32](https://github.com/OWS-PFMS/elwha/issues/32)) ‚Äî `ColorRole` (49 roles, the full M3 scheme, with `on`-pairing lookup and baseline fallback), `ShapeScale` (7), `SpaceScale` (6), `TypeRole` (12), `StateLayer` (5, with sRGB overlay blending). Every enum resolves from `UIManager` under the `Elwha.*` namespace at call time per the taxonomy's paint-time binding rule.
    - **`ElwhaTheme` install API** ([#33](https://github.com/OWS-PFMS/elwha/issues/33)) ‚Äî `Palette` (completeness-validated 49-role color map), `Theme` (`{light, dark}`), `Mode` (`LIGHT` / `DARK` / `SYSTEM` with install-time OS detection), `Typography`, `Config`, and the static `ElwhaTheme.install(Config)` facade running the locked 8-step sequence. Idempotent and re-callable ‚Äî runtime theme/mode switching is just re-installing. Every color and font is stored as a `ColorUIResource` / `FontUIResource` so `updateComponentTreeUI` correctly re-skins live components on a mode switch.
    - **FlatLaf-native key mapping** ([#34](https://github.com/OWS-PFMS/elwha/issues/34)) ‚Äî `FlatLafKeyMapping` maps a curated ~120-key subset of FlatLaf's `UIManager` keys onto token roles, and bakes M3 hover / pressed state-layer overlays into FlatLaf's discrete `*hoverBackground` / `*pressedBackground` keys so raw Swing inherits the design language. Enumerated as Appendix A of `elwha-theme-install-api.md`. The locked **focus model** is a `PRIMARY`-colored 2 px ring (`Component.focusWidth = 2`, `Component.focusColor = PRIMARY`); every `*.focused*` and `*.hover*` key is mapped explicitly to its resting equivalent ‚Äî an unmapped focus key silently falls back to FlatLaf's blue accent, so the indicator is the ring alone and the resting fill / border is preserved. `Slider.trackValueColor` (the filled portion) is mapped to `PRIMARY` so the slider's active track is on-palette.
    - **Baseline palette + JSON loader** ([#35](https://github.com/OWS-PFMS/elwha/issues/35)) ‚Äî `MaterialPalettes.baseline()` ships the M3 baseline scheme as a Elwha-normalized JSON resource loaded by `PaletteLoader`. Schema and the M3-export conversion are Appendix B of `elwha-theme-install-api.md`. Four roles the current M3 builder omits (`surfaceTint`, `background`, `onBackground`, `surfaceVariant`) are derived by canonical M3 rule.
- **Checkbox + radio split via FlatLaf icon style** ‚Äî `RadioButton.icon.style = "outlined"` separates the shared `CheckBox.icon.*` palette so the checkbox keeps the filled M3 look (primary fill, white check) while the radio renders as the canonical M3 ring (primary 2 px ring, transparent interior, primary center dot when checked). `CheckBox.icon.focusedSelectedBackground` is mapped explicitly because its fallback is `focusedBackground` (not `selectedBackground`), and the focus-ring model would otherwise make a focused, checked icon render as empty.
- **Bundled Inter font** ‚Äî the Inter Regular and Medium faces ship under `src/main/resources/com/owspfm/elwha/theme/fonts/` (SIL OFL 1.1), giving `Typography.defaults()` real M3 400/500 weight rendering. Attribution added to `NOTICE`.
- **`MaterialIcons` sized overloads** ‚Äî every lookup (`pushPin`, `delete`, ‚Ä¶) now has a sized companion (`pushPin(int size)`, etc.) so consumers don't have to chain `.derive(size, size)`.
- **`MaterialIcons.pair(name)` outline / fill helper** ‚Äî convenience for the canonical M3 toggle-icon-swap pattern. Returns a `MaterialIcons.IconPair` record with `resting()` + `filled()` accessors derived by appending `_fill` to the base name (e.g. `pair("push_pin")` loads `push_pin.svg` + `push_pin_fill.svg`). Lets `button.setIcons(p.resting(), p.filled())` be a one-liner without contaminating `ElwhaIconButton` with icon-library knowledge ‚Äî the outline/fill convention lives in the icon library that actually knows Material Symbol naming, not in the button. Sized companion `pair(name, size)` too.
- **`ThemePlayground`** ‚Äî interactive visual harness for the token foundation and the components built on it. Four top-level tabs: `Color Roles` (all 49 swatches with luminance-picked label color for non-pairing roles), `Type Scale` (the 12 type roles rendered with bundled Inter), `Swing Comps` (raw Swing widgets inheriting the theme through the FlatLaf-native key mapping, with six vertically-stacked sub-rows including a borderless M3 fill-toggle icon button and a segmented icon-toggle group), and `Chip` (an inner tabbed pane with the chip variant gallery and the live list, reused from `ChipPlaygroundPanels`). A live light / dark / system mode toggle exercises the install API's runtime-switch path.

### Changed

- **V1 `ElwhaCard` package moved to `com.owspfm.elwha.card.v1.*`** ([#83](https://github.com/OWS-PFMS/elwha/issues/83)) ‚Äî Phase 1 of epic [#80](https://github.com/OWS-PFMS/elwha/issues/80) (ElwhaCard V3 rebuild). Vacates the `card.*` namespace so V3's chrome+composition primitives can take it over starting in Phase 2 (#84+). V1 functionality is unchanged ‚Äî only the import path moves. The full tree moves: `card.ElwhaCard` / `CardVariant` / `CardInteractionMode` / `ElwhaCardDemo` ‚Üí `card.v1.*`; `card.list.*` ‚Üí `card.v1.list.*` (incl. drag-handle cursor resources under `src/main/resources/com/owspfm/elwha/card/v1/list/cursors/`); `card.playground.*` ‚Üí `card.v1.playground.*` (standalone playground exec target is now `com.owspfm.elwha.card.v1.playground.ElwhaCardPlayground`). External referrers in `chip.list.*` and `list.package-info` updated to the new path. V1 ships alongside V3 through 0.2.0 for OWS incremental migration; deleted in 1.0.0 (#96).
- **`ElwhaCard` paint pipeline ‚Äî M3-aligned shadow + crisp focus ring** ([#81](https://github.com/OWS-PFMS/elwha/issues/81)) ‚Äî `paintShadow` rewritten as a 6-layer soft-blur stack plus a key shadow directly below the card; every layer's top sits 1 px below the card top so the top silhouette stays crisp at all elevations. `shadowInsets` shrunk to match the new geometry (`top=1`, `lateral=e/2+1`, `bottom=e+2`). `paintFocusRing` now uses `BasicStroke(2f)` on a half-pixel-inset `RoundRectangle2D.Float` so the ring renders crisp instead of the 1 px draw'd ring that straddled pixel rows. Salvaged from the abandoned V2 PR stack ahead of the V3 build (epic [#80](https://github.com/OWS-PFMS/elwha/issues/80)); V1 ships with this paint until the V3 chrome class (`#84`) supersedes it.
- **`ElwhaIconButton` no longer pulls focus on mouse click** ([#81](https://github.com/OWS-PFMS/elwha/issues/81)) ‚Äî `setRequestFocusEnabled(false)` baked into the constructor. Matches M3 / macOS HIG convention for icon buttons in toolbars, app bars, and card trailing-actions slots ‚Äî focus arrives via keyboard tab navigation but not via a mouse click. Keyboard tab still focuses (gated by `setFocusable`, unchanged), so accessibility is preserved. Consumers building a "primary button" pattern can opt back in with `setRequestFocusEnabled(true)`.
- **Effective-value getters renamed to bare `getX()`** ([#62](https://github.com/OWS-PFMS/elwha/issues/62)) ‚Äî **breaking, pre-1.0, no shims.** Per the locked getter-naming rule (`docs/development/component-api-conventions.md` ¬ß1), bare `getX()` always returns the effective (resolved) value; there is no parallel `getEffectiveX()`.
    - `ElwhaChip.getEffectiveShape()` ‚Üí `getShape()`
    - `ElwhaChip.getEffectiveSurfaceRole()` ‚Üí `getSurfaceRole()`
    - `ElwhaIconButton.getEffectiveSurfaceRole()` ‚Üí `getSurfaceRole()`
    - `ElwhaSurface.getShape()` / `getSurfaceRole()` were already on the rule and are unchanged.

Decision rationale: [`docs/research/elwha-rename-decision.md`](docs/research/elwha-rename-decision.md). **Breaking** at every public surface; the lib has zero theme-API consumers and the one component consumer (OWS-Local-Search-GUI) is dev-paused, so no compat shims are provided per the pre-1.0 policy.

The icon resource path `com/owspfm/icons/material/` is unaffected.

    - **Locked-doc amendment:** [`elwha-token-taxonomy.md`](docs/research/elwha-token-taxonomy.md) and [`elwha-theme-install-api.md`](docs/research/elwha-theme-install-api.md) received a mechanical namespace amendment only (`FlatComp.*` ‚Üí `Elwha.*`, `FlatCompTheme` ‚Üí `ElwhaTheme`). The locked decisions are not reopened.
- **`MaterialIcons.DEFAULT_SIZE`: 14 ‚Üí 24** (M3 icon-button standard). The previous 14 px was chip-specialized; Material Symbols are designed at the 20-dp optical-size axis, so 24 px keeps them at their design-intended visual weight for icon buttons and toolbars. **Breaking visible change** for any consumer calling the no-arg `MaterialIcons.foo()` form ‚Äî icons will render larger by default. Internally, the 13 chip-context call sites in `ElwhaChipList` and `ElwhaChipPlayground` are pinned to an explicit `(14)` so chip visuals are preserved; chip-icon sizing will be revisited during the ElwhaChip V2 refresh.
- **`FlatPill` renamed to `ElwhaChip`** ([epic #27](https://github.com/OWS-PFMS/elwha/issues/27)) ‚Äî aligns the component name with Material Design's chip taxonomy. **Breaking change** for any consumer importing the old names, even though no consumer has shipped yet against this lib. The rename is mechanical and 1:1; no behavior changes.
    - **Package:** `com.owspfm.ui.components.pill` ‚Üí `com.owspfm.elwha.chip`; `pill.list` ‚Üí `chip.list`.
    - **Classes:** every `FlatPill*` and `Pill*` type renamed (`FlatPill` ‚Üí `ElwhaChip`, `FlatPillList` ‚Üí `ElwhaChipList`, `FlatPillPlayground` ‚Üí `ElwhaChipPlayground`, `FlatPillDemo` ‚Üí `ElwhaChipDemo`, `PillVariant` ‚Üí `ChipVariant`, `PillInteractionMode` ‚Üí `ChipInteractionMode`, `PillListModel` / `DefaultPillListModel` ‚Üí `ChipListModel` / `DefaultChipListModel`, `PillSelectionModel` / `DefaultPillSelectionModel` ‚Üí `ChipSelectionModel` / `DefaultChipSelectionModel`, `PillSelectionMode` ‚Üí `ChipSelectionMode`, `PillSelectionEvent` / `PillSelectionListener` ‚Üí `ChipSelectionEvent` / `ChipSelectionListener`, `PillAdapter` ‚Üí `ChipAdapter`, `PillReorderEvent` / `PillReorderListener` ‚Üí `ChipReorderEvent` / `ChipReorderListener`, `PillListDataEvent` / `PillListDataListener` ‚Üí `ChipListDataEvent` / `ChipListDataListener`).
    - **UIManager keys:** every `"FlatPill.*"` string value renamed to `"ElwhaChip.*"` (`ElwhaChip.background`, `ElwhaChip.hoverBackground`, `ElwhaChip.padding`, `ElwhaChip.arc`, `ElwhaChip.warmAccent`, `ElwhaChip.foreground`, ‚Ä¶). Java constant names (`K_*`, `STYLE_PROPERTY`) are unchanged; only the string values move. The client-property key `"FlatPill.style"` is now `"ElwhaChip.style"`, and the ad-hoc `"FlatPill.removeIcon"` override slot used by the playground is now `"ElwhaChip.removeIcon"`.
    - **Consumer migration:** any consumer that previously overrode `UIManager.put("FlatPill.*", ...)` at app startup must update its overrides to the `ElwhaChip.*` namespace. Imports of `com.owspfm.ui.components.pill.*` must update to `com.owspfm.elwha.chip.*`. Tracked downstream in [OWS-PFMS/OWS-Local-Search-GUI#258](https://github.com/OWS-PFMS/OWS-Local-Search-GUI/issues/258).
    - **History:** the component was originally named `FlatPill` because its shape is a capsule ("pill"); the Material industry-standard name is "chip" for interactive token elements (input / choice / filter / action chips). Pre-flight inventory of every renamed touchpoint lives at `docs/research/flatpill-to-flatchip-rename-inventory.md`.
- **Naming convention switched to Google Java Style** ([epic #5](https://github.com/OWS-PFMS/elwha/issues/5)). Every `my*`-prefixed instance field and every `the*`-prefixed parameter / local was renamed to its unprefixed equivalent across the source tree. Public API parameter names move with this change ‚Äî Java does not bind to parameter names at the call site, but Javadoc `@param` references and consumers reading reflection metadata will see the new names. No method signatures, class names, package names, or behavior changed.
- **`ChipVariant.GHOST` does not participate in selection rendering** ([#50](https://github.com/OWS-PFMS/elwha/issues/50), rebuild doc ¬ß9 Q2 amendment) ‚Äî GHOST is Elwha's equivalent of M3's Text button (the lowest of M3's five emphasis levels), and M3 doesn't define a selected state at that emphasis level. Resolves a visual collapse surfaced during PR #46 smoketest: under the preemptive `PRIMARY` border swap on selected, `GHOST + SELECTED` and `OUTLINED + SELECTED` rendered identically (both: no fill + 12 % overlay + 1 px `PRIMARY` border), and `GHOST + SELECTED` additionally read as a faded `FILLED` chip ‚Äî defeating GHOST's "no surface unless interacting" intent. The `selected` field and `PROPERTY_SELECTED` events are preserved on the API surface (so `setVariant(...)` later to a non-GHOST variant resumes rendering the selected state), but the rendering pipeline ignores `selected` while variant is `GHOST` ‚Äî no `SELECTED` state-layer overlay, no border swap. Consumers needing a togglable chip with a visible selected state should use `OUTLINED` (M3's default treatment for filter chips). `ChipVariant √ó ChipInteractionMode` orthogonality is preserved at the API level ‚Äî only the visual contract changes.

### Removed

- **`ChipVariant.WARM_ACCENT`** (epic #31) ‚Äî never an M3 concept and bundled a color choice into what should be a treatment axis. The secondary-container look is now `FILLED` + `setSurfaceRole(SECONDARY_CONTAINER)` ‚Äî color and treatment cleanly orthogonal.
- **`ElwhaChip.*` UIManager namespace** (epic #31) ‚Äî the entire ~12-key namespace (`ElwhaChip.background`, `borderColor`, `arc`, `padding`, `hoverBackground`, `pressedBackground`, `selectedBackground`, `selectedBorderColor`, `focusColor`, `disabledBackground`, `warmAccent`, `foreground`) and the `K_*` public constants are deleted. App-wide chip theming now happens through the installed palette + the variant's role declarations; per-instance happens through the typed setters. The `"ElwhaChip.style"` client-property escape hatch is also gone ‚Äî typed setters cover what it did.
- **`ElwhaChip` raw-typed setters** (epic #31) ‚Äî `setSurfaceColor(Color)`, `setForegroundColor(Color)`, `setCornerRadius(Integer)`, `setBorderColor(Color)`, `setPadding(Insets)`, `setPadding(int, int)`. Replaced by `setSurfaceRole(ColorRole)`, `setShape(ShapeScale)`, `setPadding(SpaceScale, SpaceScale)`. `setBorderWidth(int)` is **kept** ‚Äî stroke width is genuine geometry with no token equivalent.
- **`ElwhaChip` foreground override** (epic #31) ‚Äî no per-instance foreground setter. Foreground is always the `on`-pair of the effective surface role, so an independent override would re-introduce unpaired surface/foreground.
- **`ElwhaChip` auto-contrast machinery** (epic #31) ‚Äî `resolveForegroundColor()`'s luma branch, `AUTO_FG_DARK` / `AUTO_FG_LIGHT`, `isLight`, `effectiveSurfaceForContrast`. The token system's `on`-pairing makes them unnecessary.
- **`ElwhaChipDemo`** (epic #31) ‚Äî the minimal smoke-test demo. Redundant with `ElwhaChipPlayground` after the playground rework; the playground now covers the same matrix plus the live-list surface.

### Fixed

- **`SurfacePainter` stroke geometry for borders wider than 1 px** (surfaced by smoketesting [#43](https://github.com/OWS-PFMS/elwha/issues/43)) ‚Äî the painter centered the stroke on a path inset by `0.5 px` (correct for a 1 px AA line on the half-pixel grid), so any wider stroke extended outside the component bounds and got clipped: at width 8 only ~4.5 px of fuzzy inner stroke was visible instead of a crisp outer ring. Fixed by insetting the stroke centerline by `borderWidth / 2` and shrinking the corner diameter by `borderWidth` so the stroke's outer edge tracks the fill's corner exactly. Fill now extends to the full component bounds for clean AA at the corners. Latent under `ElwhaChip` (max stroke 2 px, artifact below the visibility threshold); newly user-visible via `ElwhaSurface.setBorderWidth(int)`.
- **`ElwhaChip` text baseline drift when leading affordance is hidden** ([#49](https://github.com/OWS-PFMS/elwha/issues/49)) ‚Äî chips with no visible leading affordance but a visible trailing icon read with the text sitting ~2px high relative to the trailing icon's centerline. Root cause: stock `FlowLayout` skipped invisible children for row-height computation, so the leading cluster reported a shorter preferred height than the trailing button's hit-target; when `BorderLayout` stretched the cluster to match, the slack landed below the row instead of being split above and below. Fixed by an internal `BaselineCenteringFlowLayout` that floors the row height at `BUTTON_MIN_HIT_TARGET` and re-centers the row after the parent's layout pass. Predates epic #31; surfaced by smoketesting the chip rebuild.
- **Bundled `MaterialPalettes.baseline()` regenerated from a fresh Material Theme Builder export** ([#47](https://github.com/OWS-PFMS/elwha/issues/47)) ‚Äî the prior baseline scheme produced muddy / low-legibility chip surfaces, and additionally collapsed the secondary role onto primary (`schemes.light.primary == schemes.light.secondary`), nullifying the primary/secondary emphasis hierarchy M3 components are designed around. New palette is a clean single-seed M3 scheme (`#6750A4`); secondary and tertiary tonal palettes are derived by canonical hue-rotation, so `primary` / `secondary` / `tertiary` (and their containers) are distinct in both light and dark. All 49 `ColorRole` keys now come directly from `schemes.light` / `schemes.dark` (no derived roles). Raw export archived at `docs/research/baseline-palette-mtb-export-2026-05-15.json`.

Note: requires JDK 21 ‚Äî Spotless + google-java-format hits a binary-API drift on JDK 22+ (tracked upstream at [diffplug/spotless#2468](https://github.com/diffplug/spotless/issues/2468)).

## [0.1.0] ‚Äî 2026-05-12

Initial release. Library extracted from the [OWS-Local-Search-GUI](https://github.com/OWS-PFMS/OWS-Local-Search-GUI) project ([epic #231](https://github.com/OWS-PFMS/OWS-Local-Search-GUI/issues/231)).

### Added

- **`ElwhaCard`** ‚Äî theme-aware card primitive with header, body, surface variants, hover/pressed/selected states, optional collapse/expand, leading icon, trailing actions.
- **`ElwhaCardList<T>`** ‚Äî list of `ElwhaCard` items: selection, drag-to-reorder, filter, sort, orientation modes (vertical / horizontal / wrap / grid).
- **`FlatPill`** ‚Äî compact pill primitive: text + optional leading icon + optional trailing action button, themeable via UIManager keys (`FlatPill.*`), auto-contrast foreground via BT.601 luma.
- **`FlatPillList<T>`** ‚Äî list of `FlatPill` items with selection modes (`NONE` / `SINGLE` / `SINGLE_MANDATORY` / `MULTIPLE`), drag-to-reorder, pinned-partition + anchored modes (`MovementMode` enum), icon affordances (`IconAffordance` enum).
- **`ElwhaList<T>`** ‚Äî shared cross-cutting interface implemented by both `ElwhaCardList` and `FlatPillList` (orientation, gap, padding, empty / loading state, filter, sort).
- **`MaterialIcons`** ‚Äî helper that exposes 17 Material Symbols SVGs (Rounded / 400 / fill0 / 20px) via `FlatSVGIcon` with a theme-aware color filter.
- **`ElwhaCardPlayground`** ‚Äî interactive playground for the card family (embeds `ElwhaCardListShowcase` + `GalleryPanel` panels for the list view, single-card variants, and cycle/cursor examples).
- **`FlatPillPlayground`** ‚Äî interactive playground for the pill list family, including a LAF tweak panel for live UIManager-key experimentation.

### Notes

- API is **not yet stable** at 0.1.0. Breaking changes between minor versions are expected until 1.0.0.
- Two breaking-change refactors are tracked and gated on this release:
  - `FlatPill` ‚Üí `ElwhaChip` rename (taxonomy alignment with Material)
  - `ElwhaList<T>` extension to share selection + drag-reorder surface across both list families
- 1.0.0 will follow those landings.
